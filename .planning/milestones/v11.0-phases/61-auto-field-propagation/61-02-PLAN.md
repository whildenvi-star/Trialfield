**farm-budget/server.js — Modify POST /api/fields (line ~476):**

Add a duplicate guard at the top of the handler, before creating the field:
1. If `req.body.registryFieldId` is provided, check `store.fields.find(f => f.registryFieldId === req.body.registryFieldId)`.
2. If a match is found, return `res.status(200).json(enrichField(existingField))` — idempotent response, no duplicate created.
3. If no match, proceed with existing creation logic. The existing `Object.assign({ id: generateId('fld') }, req.body)` already copies `registryFieldId` and `registryFieldName` from the body, since `registryFieldId` is already in the updatable fields list (confirmed at line ~496). No changes needed to the creation logic itself — just the duplicate guard.

**grain-tickets/server.js — Modify POST /api/farms (line ~1197):**

Add a duplicate guard at the top of the handler:
1. If `req.body.registryId` is provided, check for existing farm: `await prisma.farm.findFirst({ where: { registryId: req.body.registryId } })`.
2. If found, return `res.status(200).json(dbFarmToJson(existingFarm))` — idempotent.
3. If no match by registryId, also check by name: `await prisma.farm.findFirst({ where: { name: (req.body.farm || '').trim() } })`.
4. If name match found AND it has no registryId, update it with the registryId: `await prisma.farm.update({ where: { id: existingFarm.id }, data: { registryId: req.body.registryId } })` and return 200 with the updated farm.
5. If name match found AND it already has a different registryId, return 200 with existing (don't overwrite).
6. Only if no matches at all, proceed with existing creation logic (which already accepts `registryId` in the create data).

Important: The existing P2002 (unique constraint) catch on grain-tickets handles the `name` unique constraint. The duplicate guard by name in step 3-5 is a softer pre-check that also wires the registryId on existing records.

  <name>Task 2: Create portal webhook endpoint for auto CLU record creation</name>
  <files>glomalin-portal/src/app/api/fsa/webhook/field-created/route.ts</files>
  <action>
Create a new Next.js API route at `glomalin-portal/src/app/api/fsa/webhook/field-created/route.ts`.

This endpoint receives webhook calls from farm-registry when a new field is created. It creates a minimal CLU record with the registry_field_id pre-wired.

**POST handler:**
1. **Auth:** Check for a `token` query parameter matching `process.env.EMBED_TOKEN`. If no EMBED_TOKEN is set, skip auth (dev mode). If token doesn't match, return 403.
   - Use `new URL(request.url).searchParams.get('token')` to read the token.
   - Do NOT use the Supabase guard here — this is a machine-to-machine webhook, not a user request.
2. **Parse body:** Read `{ field_name, fsa_acres, registry_field_id }` from request JSON.
3. **Validate:** `field_name` and `registry_field_id` are required. Return 400 if missing.
4. **Duplicate check:** Query Supabase for existing clu_records with matching `registry_field_id` in the current crop year. Use `createClient` from `@supabase/supabase-js` with `process.env.NEXT_PUBLIC_SUPABASE_URL` and `process.env.SUPABASE_SERVICE_ROLE_KEY` (service role — no RLS needed for webhook).
   - If found, return `{ status: 200, record: existingRecord }` — idempotent.
5. **Create CLU record:** Insert into clu_records:
   ```
   {
     field_name: field_name,
     farm_name: field_name,
     fsa_acres: fsa_acres || 0,
     registry_field_id: registry_field_id,
     crop_year: CURRENT_CROP_YEAR (import from @/lib/config),
     farm_number: 0,    // placeholder — user fills in real FSA numbers later
     tract_number: 0,   // placeholder
     clu: field_name,   // use field name as CLU identifier placeholder
     reported: false,
     irrigated: false,
     organic: false,
     double_crop: false,
     cover_crop: false,
     prevented_planting: false,
   }
   ```
6. Return 201 with `{ record: data }`.

**Import requirements:**
- `import { createClient } from '@supabase/supabase-js'`
- `import { CURRENT_CROP_YEAR } from '@/lib/config'`
- `import { NextResponse } from 'next/server'`

**Important:** Use service role key for the Supabase client — this bypasses RLS since the webhook is a trusted server-to-server call. The service role key is already available as `SUPABASE_SERVICE_ROLE_KEY` in the portal's .env.
  </action>
