<objective>
Build the sync orchestration service and all Phase 1 API routes. The sync service coordinates fetching data from Case IH (via fieldops-client), normalizing it (via fieldops-normalizer), writing staging rows (SyncedOperation), and updating sync state (FieldOpsSyncState). The API routes expose ADMIN-gated endpoints for triggering sync, managing field mappings, managing operation type mappings, reviewing/approving/rejecting staged operations, retrieving sync state, and disconnecting the Case IH connection.

Purpose: This is the backend engine of Phase 1. Without these routes, the UI pages in Plan 03 have nothing to call. The sync service implements the core data pipeline: Case IH API -> normalize -> stage -> review -> commit. The sync-state and connection endpoints ensure the UI can load current state on mount and allow the admin to disconnect/reconnect.

Output: fieldops-sync.ts service, 9 API route files covering sync trigger, field matching, operation type mapping, staged operation review, sync state retrieval, and connection disconnect.
</objective>

  <files>organic-cert/src/lib/fieldops-sync.ts</files>
  <action>
Create `organic-cert/src/lib/fieldops-sync.ts` that orchestrates the full sync pipeline.

Export `runFieldOpsSync(farmId: string)` which:

1. **Generate sync run ID**: Create a unique syncRunId (use cuid or Date.now-based ID).

2. **Validate connection (API-05)**:
   - Call `validateConnection(farmId)` from fieldops-client.
   - If linkedAccountWarning is true, update FieldOpsSyncState with `lastSyncStatus: 'no_data'`, `linkedAccountWarning: true`, return early with `{ status: 'no_data', linkedAccountWarning: true, message: 'Case IH account returned no fields. This may be a Linked Account limitation.' }`.

3. **Load field mappings**: Query `prisma.caseIHFieldMapping.findMany({ where: { farmId } })` to get existing field-to-field mappings.

4. **Load operation type mappings**: Query `prisma.operationTypeMapping.findMany({ where: { farmId } })`.

5. **Fetch Case IH data for 3-year lookback** (per user decision: initial sync pulls last 3 years):
   - Calculate seasons: current year, current year - 1, current year - 2 (use `date-fns` for year calculation).
   - For each season, call `getApplications(undefined, season)` and `getYield(undefined, season)`.
   - Collect all raw application and yield records.

6. **Normalize data**:
   - Call `normalizeApplications(rawApplications, fieldMappings, opTypeMappings)` from fieldops-normalizer.
   - Call `normalizeYield(rawYield, fieldMappings)` from fieldops-normalizer.
   - Collect all staged records and all warnings.

7. **Write staging rows**:
   - For each staged record, use `prisma.syncedOperation.upsert()` with `where: { farmId_fieldopsExternalId: { farmId, fieldopsExternalId } }`.
   - On upsert conflict (re-sync): only update PENDING rows. Skip rows that are already APPROVED or REJECTED (per pitfall #4 in research).
   - Set syncRunId on all upserted rows.

8. **Update sync state**:
   - Upsert `prisma.fieldOpsSyncState` for this farmId with: lastSyncAt: now, lastSyncStatus: (warnings.length > 0 ? 'partial' : 'success'), lastSyncRunId, totalFieldsMapped (count of field mappings), totalUnmatched (count of Case IH fields not in mappings).

9. **Return sync result**: `{ status, syncRunId, operationsStaged, operationsSkipped (already approved/rejected), warnings, fieldsProcessed, unmatchedFields[] }`.

IMPORTANT:
- Do NOT commit staged operations to FieldOperation/HarvestEvent tables. That happens in the review/approve flow (staged-ops API route).
- Use `prisma.$transaction()` for the batch upsert to ensure atomicity.
- Catch errors per-field (one failed field should not abort the entire sync). Collect errors in warnings array.

Create 8 API route files, all ADMIN-gated using the existing auth pattern from `organic-cert/src/lib/auth.ts`. Follow the existing route pattern in `organic-cert/src/app/api/fields/route.ts` for consistent error handling and response format.

Every route handler must:
- Call `auth()` from `@/lib/auth` to get session
- Check `(session?.user as any)?.role === 'ADMIN'` -- return 403 if not ADMIN
- Use `(session?.user as any)?.farmId` for all Prisma queries (farm-scoped data)
- Return NextResponse.json with appropriate status codes
- Wrap in try/catch, return 500 with error message on failure

**Route 1: POST /api/admin/sync/route.ts**
- Imports `runFieldOpsSync` from `@/lib/fieldops-sync`.
- Calls `runFieldOpsSync(farmId)`.
- Returns the sync result JSON (status, operationsStaged, warnings, etc.).
- If sync throws, returns 500 with error message.

**Route 2: GET /api/admin/fieldops/fields/route.ts**
- Calls `getFields()` from `@/lib/fieldops-client`.
- Also loads existing `CaseIHFieldMapping` records for this farm.
- Returns `{ caseIHFields: [...], existingMappings: [...] }` so the UI can show which fields are already matched.

**Route 3: GET/POST/PUT /api/admin/fieldops/mappings/route.ts**
- GET: Returns all CaseIHFieldMapping records for this farm, joined with organic-cert field names.
- POST: Creates a new mapping. Body: `{ caseIHFieldId, caseIHFieldName, caseIHFarmName?, organicCertFieldId }`. Uses upsert on `farmId_caseIHFieldId` to handle re-mapping.
- PUT: Updates an existing mapping. Body: `{ id, organicCertFieldId }`.

**Route 4: GET/POST /api/admin/fieldops/op-types/route.ts**
- GET: Returns all OperationTypeMapping records for this farm. Also returns a list of unmapped rawOpTypes (found in SyncedOperation but not in OperationTypeMapping).
- POST: Creates a new mapping. Body: `{ rawOpType, nopCategory }`. Uses upsert on `farmId_rawOpType`.

**Route 5: GET /api/admin/fieldops/sync-state/route.ts**
- Returns the FieldOpsSyncState record for this farm: `prisma.fieldOpsSyncState.findUnique({ where: { farmId } })`.
- Response shape: `{ lastSyncAt, lastSyncStatus, lastSyncRunId, totalFieldsMapped, totalUnmatched, lastSyncError, linkedAccountWarning, createdAt, updatedAt }`.
- If no FieldOpsSyncState record exists for this farm, return `{ connected: false }` with 200 status (not 404 -- the UI uses this to determine connection state).
- If a record exists, include `connected: true` in the response.
- This endpoint is called by the FieldOps hub page on mount to display current sync status.

**Route 6: DELETE /api/admin/fieldops/connection/route.ts**
- Disconnects the Case IH integration for this farm.
- Deletes the FieldOpsSyncState record for this farm.
- Deletes all CaseIHFieldMapping records for this farm.
- Deletes all OperationTypeMapping records for this farm.
- Does NOT delete SyncedOperation records (they are historical audit trail -- keep for compliance).
- Uses `prisma.$transaction()` to ensure atomicity of the three deletes.
- Returns `{ disconnected: true, message: 'Case IH connection removed. Field mappings and operation type mappings have been cleared. Previously synced operations are preserved for audit history.' }`.
- Per locked user decision: "Disconnect/reconnect option available in settings."

**Route 7: GET /api/admin/staged-ops/route.ts**
- Returns SyncedOperation records for this farm.
- Query params: `status` (filter by PENDING/APPROVED/REJECTED, default PENDING), `syncRunId` (filter by specific sync run).
- Includes mappedFieldId resolved to field name if available.
- Paginate: `page` and `limit` query params (default limit 50).

**Route 8: POST /api/admin/staged-ops/[id]/route.ts**
- Body: `{ action: 'approve' | 'reject', rejectedReason?: string }`
- On APPROVE:
  1. Load the SyncedOperation record. Must be PENDING (return 400 if already approved/rejected).
  2. Check for existing manual records: query FieldOperation or HarvestEvent with matching fieldEnterpriseId + operationDate + type. If manual record exists, return 409 with message "A manual record already exists for this date and operation type. Manual data takes priority per farm policy." (per user decision: manual always wins).
  3. If no conflict: create the appropriate domain record (FieldOperation for TILLAGE/PLANTING/CULTIVATION types, HarvestEvent for yield/harvest types). Set the SyncedOperation status to APPROVED, reviewedAt to now, reviewedByUserId to session user id.
  4. Log the action via audit-logger (logAudit) for compliance tracking.
- On REJECT: Update SyncedOperation status to REJECTED with rejectedReason, reviewedAt, reviewedByUserId.

IMPORTANT: For the approve flow, the mapping from SyncedOperation to domain models (FieldOperation, HarvestEvent) requires a `fieldEnterpriseId`. The SyncedOperation has `mappedFieldId` which is a Field.id. You need to look up the active FieldEnterprise for that field to get the fieldEnterpriseId. If no FieldEnterprise exists for the mapped field, return 400 with a helpful message ("No active crop enterprise found for this field. Create a field enterprise first.").
