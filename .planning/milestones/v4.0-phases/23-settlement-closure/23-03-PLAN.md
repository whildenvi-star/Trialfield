Enhance the dispute resolution workflow with proper resolution statuses, notes, and dates. Add a multi-buyer season summary view that shows all buyers for a crop year in a single consolidated view.

Purpose: The existing dispute feature only sets matchStatus to 'disputed' with a free-text note — there's no structured resolution workflow.   <name>Task 1: Add resolution fields to SettlementLine, enhance dispute API, and create season summary endpoint</name>
  <files>grain-tickets/prisma/schema.prisma, grain-tickets/server.js</files>
  <action>
1. **Schema migration** — Add three new fields to the `SettlementLine` model in `grain-tickets/prisma/schema.prisma`:
   - `resolutionStatus String?` — One of: "Buyer Error", "Our Error", "Write-off", "Pending", or null (not disputed).
   - `resolutionNotes String?` — Free-text resolution notes (separate from the existing `notes` field which holds general line notes).
   - `resolutionDate DateTime?` — Date the dispute was resolved.
   Run `npx prisma migrate dev --name add-resolution-fields` from `grain-tickets/` directory.

2. **Enhance `PATCH /api/settlement-lines/:lineId/dispute`** (around line 1809 in server.js):
   - Accept additional body fields: `resolutionStatus` (required, must be one of "Buyer Error", "Our Error", "Write-off", "Pending"), `resolutionNotes` (optional string), `resolutionDate` (optional ISO date string).
   - Validate resolutionStatus against the allowed values. Return 400 if invalid.
   - Update the SettlementLine with matchStatus='disputed' plus the new resolution fields.
   - If resolutionStatus is "Buyer Error", "Our Error", or "Write-off" and resolutionDate is not provided, auto-set resolutionDate to now().
   - If resolutionStatus is "Pending", clear resolutionDate (set to null) since it's not yet resolved.
   - Keep backward compatibility: if only `notes` is sent (no resolutionStatus), behave as before (set matchStatus='disputed', save notes to the existing notes field).

3. **Add `GET /api/reconciliation/season-summary?cropYear=N`** endpoint:
   - Fetches a cross-buyer summary for the entire crop year.
   - For each buyer that has either tickets or settlements in that crop year:
     a. `buyerId`, `buyerName`, `buyerShortCode`
     b. `ticketCount` — Total farm tickets sent to this buyer in the crop year
     c. `totalWeightLbs` — Sum of farm ticket netWeight
     d. `settlementLineCount` — Total settlement lines imported for this buyer+cropYear
     e. `matchedCount` — Lines with matchStatus 'matched' or 'manual'
     f. `unmatchedCount` — Lines with matchStatus 'unmatched'
     g. `disputedCount` — Lines with matchStatus 'disputed'
     h. `totalPayment` — Sum of netPayment across all settlement lines (Decimal → number). Handle null netPayment gracefully.
     i. `varianceLbs` — Total farm weight minus total buyer weight (from matched lines only)
     j. `variancePct` — varianceLbs as percentage of farm weight
     k. `paymentStatus` — Derived: "No Settlements" if no settlements exist, "Fully Matched" if unmatchedCount=0 and disputedCount=0, "Has Disputes" if disputedCount > 0, "Partially Matched" if unmatchedCount > 0
   - Sort by buyerName ascending.
   - Return as a JSON array.

   **Implementation approach:** Use two main queries:
   - `prisma.ticket.groupBy({ by: ['buyerId'], where: { cropYear }, _count: true, _sum: { netWeight: true } })` for farm-side aggregation.
   - `prisma.settlement.findMany({ where: { cropYear }, include: { buyer: true, lines: true } })` for settlement-side aggregation.
   - Join in JS by buyerId. Also fetch buyers with `prisma.buyer.findMany()` to get names for buyers that may have tickets but no settlements yet.
