<objective>
Add fuzzy settlement matching that automatically searches for candidate matches when exact ticket number matching fails, using date proximity (within 2 days) and weight similarity (within configured tolerance), presenting candidates for user confirmation.

Purpose: Real-world ticket numbers often don't match exactly between farm records and buyer settlements — different prefixes, typos, or completely different numbering systems. Fuzzy matching by date and weight gives the user a realistic way to close out unmatched lines without tedious manual cross-referencing.

Output: Fuzzy candidate search API endpoint, candidate display in unmatched panels, confirmation-to-link flow.
</objective>

  <name>Task 1: Create fuzzy candidate search endpoint</name>
  <files>grain-tickets/server.js</files>
  <action>
1. **Add `GET /api/reconciliation/fuzzy-candidates`** endpoint in `grain-tickets/server.js`, near the existing reconciliation endpoints (after `/api/reconciliation/unmatched` around line 1775).

   **Query params:** `buyerId` (required), `cropYear` (required).

   **Logic:**
   - Fetch all unmatched settlement lines for this buyer+cropYear (lines with matchStatus='unmatched' across settlements for that buyer+cropYear).
   - Fetch all farm tickets for this buyer+cropYear that are NOT already matched to any settlement line (i.e., no settlementLine with matchStatus 'matched' or 'manual' pointing to them).
   - For each unmatched settlement line, search for candidate farm tickets where:
     a. **Date proximity:** Farm ticket date is within +/- 2 calendar days of the settlement line's date. Handle null dates gracefully (skip date filter if either is null).
     b. **Weight proximity:** Farm ticket netWeight is within the configured tolerance for that crop. Look up CropConfig for the ticket's crop+cropYear to get tolerancePct/toleranceLbs (from Plan 23-01). If tolerance is 0 (not configured), use a default 2% for fuzzy search purposes.
     c. Compute `weightVarianceLbs = Math.abs(farmTicket.netWeight - line.netWeight)` and `weightVariancePct = (weightVarianceLbs / farmTicket.netWeight) * 100`.
   - Return an array of objects, one per unmatched settlement line that has at least one candidate:
     ```json
     {
       "settlementLineId": 123,
       "lineTicketNo": "H066666",
       "lineDate": "2025-09-15",
       "lineNetWeight": 55480,
       "candidates": [
         {
           "ticketId": 456,
           "ticketNo": "066667",
           "date": "2025-09-14",
           "netWeight": 55520,
           "crop": "Hybrid Rye",
           "weightVarianceLbs": 40,
           "weightVariancePct": 0.07,
           "dateDiffDays": 1
         }
       ]
     }
     ```
   - Sort candidates within each line by smallest weight variance first, then by smallest date diff.
   - Limit candidates to 5 per settlement line (avoid overwhelming the UI).

2. **Performance:** Use a single query for farm tickets and a single query for settlement lines, then do the matching in JS. Do NOT do N+1 queries per line. The dataset is small (100-500 tickets/season) so in-memory matching is fine.

3. **Date math helper:** Create a simple `daysDiff(d1, d2)` helper that returns the absolute number of calendar days between two dates. Handle timezone by comparing date-only strings (YYYY-MM-DD).
