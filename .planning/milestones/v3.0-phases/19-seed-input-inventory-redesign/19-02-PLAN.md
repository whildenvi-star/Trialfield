<objective>
Build the Forecast Hub and Orders tab UI — the core procurement command center.

Purpose: The Forecast Hub is the "home tab" of the redesigned inventory system — it shows farm-wide procurement needs computed live from Macro Roll-Up data, with visual progress bars showing ordering status. The Orders tab manages purchase orders created from forecast selections. Together these two modules deliver the primary user workflow: see what you need, order it.

Output: Fully functional Forecasts tab with expandable category tables, checkbox selection, and Create Order flow. Fully functional Orders tab with list, detail, edit, and status filtering.
</objective>

  <name>Task 1: Forecast Hub tab — category tables, expandable rows, checkboxes, status bars</name>
  <files>farm-budget/public/inventory.js</files>
  <action>
Replace the placeholder IIFE shell in `inventory.js` with the full Forecast Hub implementation. Follow the existing vanilla JS IIFE pattern used by `inputs-manager.js` and `seed-manager.js`.

**Tab activation:** Listen for `tab-activate` with `e.detail.tab === 'forecasts'`. On activation, always call `loadForecast()` (set `loaded = false` — forecast is live, never cached).

**Forecast-changed listener:** Also listen for the `forecast-changed` custom event (dispatched by deliveries.js after a delivery save). If the forecasts tab is currently active (i.e., `document.getElementById('tab-forecasts').classList.contains('active')`), call `loadForecast()` to refresh the status bars in-place without requiring tab re-navigation:
```javascript
window.addEventListener('forecast-changed', function() {
  if (document.getElementById('tab-forecasts').classList.contains('active')) {
    loadForecast();
  }
});
```

**loadForecast():** Call `api.get('/api/forecast')`, store result in module-level `currentForecast`. Then call `renderForecast(data)`.

**renderForecast(data):**
1. Target container: `document.getElementById('fc-categories')`
2. For each category in `data.categories`, render a collapsible section:
   - Category header: `<h3>` with category name and product count badge
   - Table with columns: `[ ] | Product | Supplier | Amount | Unit | Unit Cost | Total Cost | Ordered | Remaining | % Ordered`
   - The checkbox column contains `<input type="checkbox">` per row for order creation
   - The `% Ordered` column shows a `.pct-bar` div with `.pct-bar-fill` width set to `pctOrdered + '%'`. Add class `.complete` if pctOrdered >= 100, `.over` if pctOrdered > 100.
   - Amount column shows `util.formatNum(totalQty, 0)`, Unit Cost shows `util.formatMoney(unitCost, 4)`, Total Cost shows `util.formatMoney(totalCost, 2)`
   - Ordered shows `util.formatNum(orderedQty, 0)`, Remaining shows `util.formatNum(remaining, 0)` with red text class if negative

3. **Expandable field breakdown:** Each product row has a click handler (on the product name cell) that toggles a detail row below it. The detail row contains a sub-table: `Field | Acres | Qty | Season`. Use CSS `max-height` transition for expand/collapse (existing app pattern). Use a caret/chevron indicator on the product name.

4. **Only products with amount > 0 appear** — the server already filters this, but add a client-side guard too.

5. **Summary line:** Update `#fc-summary` with total product count and total cost across all categories: e.g., "64 products — $142,500 total forecast"

**Checkbox selection state:**
- Module-level `var selectedProducts = new Set()` keyed by productName
- Each checkbox `onchange` adds/removes from the set
- Enable/disable `#fc-create-order` button based on `selectedProducts.size > 0`
- Add "Select All" / "Clear" quick actions per category header

**Create Order button handler (`#fc-create-order`):**
1. Gather selected products from `currentForecast.categories.flatMap(c => c.products).filter(p => selectedProducts.has(p.productName))`
2. Group by `supplierId` (use `'__none__'` for empty supplierId)
3. For each supplier group, call `api.post('/api/orders', { supplierId, supplierName, status: 'ordered', poNumber: '', notes: '', createdAt: new Date().toISOString(), items: [...] })` where each item has `{ productName, unit, forecastQty: totalQty, orderedQty: totalQty, unitCost }`
4. After all orders created, show toast `"Created N order(s)"`, clear selection, reload forecast (to update ordered quantities), and switch to Orders tab: `location.hash = 'orders'`

**Print Reports dropdown (`#fc-print-menu`):** Create a dropdown menu (click to toggle, click outside to close) with 5 report options. Each option dispatches a custom event `window.dispatchEvent(new CustomEvent('print-report', { detail: { type: 'agronomist' } }))` which reports.js (Plan 03) will handle. For now, wire the dropdown menu structure and event dispatch — actual report rendering is in Plan 03.

Report menu items:
- Agronomist/Supplier Order Sheet
- Field-Level Input Plan
- Forecast Summary
- Order Status Report
- Delivery Receipt Log

**Empty state:** If forecast has 0 categories or 0 total products, show `util.emptyState('📦', 'No forecast data', 'Add inputs to fields in the Macro Roll-Up to see forecasts here')`.

**Touch-friendly:** Ensure checkboxes and expand targets are at least 44x44px tap targets. Use `padding` on cells, not tiny inline elements.

**Tab activation:** Listen for `tab-activate` with `e.detail.tab === 'orders'`. On activation, call `loadOrders()`.

**loadOrders():** Call `api.get('/api/orders')`, store in module-level `allOrders`. Apply status filter from `#ord-filter-status`, then render.

**renderOrderList(orders):**
1. Target container: `document.getElementById('ord-list')`
2. Render each order as a card (`.order-card`) with:
   - Header: Supplier name, PO# (if set), status badge (color-coded: ordered=blue, partial=amber, complete=green)
   - Created date formatted with `new Date(order.createdAt).toLocaleDateString()`
   - Item count summary: "N items — $X total"
   - Click to expand inline detail (same expand pattern as forecast)

3. **Expanded order detail** shows:
   - Editable PO Number field (input, blur saves via `api.put`)
   - Editable Notes field (textarea, blur saves)
   - Line items table: Product | Unit | Forecast Qty | Ordered Qty | Delivered Qty | Remaining
     - Ordered Qty is editable (double-click inline edit pattern from inputs-manager.js)
     - Delivered Qty is computed from `store.deliveries` — the server handles this, so fetch fresh order data after delivery changes
     - Remaining = orderedQty - deliveredQty (red if negative)
   - "Record Delivery" button — switches to Deliveries tab with orderId pre-selected: `location.hash = 'deliveries'; window.dispatchEvent(new CustomEvent('start-delivery', { detail: { orderId: order.id } }))`
   - "Delete Order" button with confirmation (`confirm()` dialog)

4. **Status filter:** `#ord-filter-status` select change triggers re-render with filtered list.

5. **Order count:** Update `#ord-count` with "N orders" or "N of M orders" when filtered.

6. **Empty state:** `util.emptyState('📋', 'No orders yet', 'Select products on the Forecasts tab and click Create Order')`.

7. **Sort:** Most recent orders first (by `createdAt` descending).

**Delivery quantity aggregation:**
To show delivered quantities per order line, add a helper that calls `api.get('/api/deliveries?orderId=' + order.id)` when expanding an order, then sums `deliveredQty` per productName across all deliveries for that order. Cache per order expansion session.

**Order card CSS** (add to style.css or inline in the module):
- `.order-card` with card background, border-radius, padding consistent with existing `.prog-card` styling
- Status badge colors: `.status-ordered { color: var(--primary); }`, `.status-partial { color: var(--amber); }`, `.status-complete { color: var(--success); }`
