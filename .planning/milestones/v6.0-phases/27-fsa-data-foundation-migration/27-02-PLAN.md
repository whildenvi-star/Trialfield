**1. Create GET /api/fsa/validation/route.ts**

Follow the existing Route Handler pattern from `api/admin/users/route.ts`:

```typescript
import { NextResponse } from 'next/server'
import { createClient } from '@/lib/supabase/server'
import { validateCluRecords } from '@/lib/fsa/calc'
```

Handler logic:
1. Auth check: `const { data: { user } } = await supabase.auth.getUser()` — return 401 if no user
2. Fetch 3 datasets in parallel using `Promise.all`:
   - `supabase.from('clu_records').select('*').eq('crop_year', 2026)`
   - `supabase.from('insurance_pricing').select('*').eq('year', 2026)`
   - `supabase.from('insurance_policies').select('*').eq('policy_year', 2026)`
3. Call `validateCluRecords(cluRecords, pricing, policies)` with the fetched data (using `?? []` for null safety)
4. Return `NextResponse.json({ warnings, recordCount: cluRecords?.length ?? 0 })`
5. If any Supabase query errors, return 500 with error message

The `year` could be parameterized via searchParams but default to 2026 for now (hardcoded is fine — Phase 28 will add a year selector).

**2. Create GET /api/fsa/auto-populate-preview/route.ts**

This endpoint fetches farm-budget dashboard data and generates crop assignment proposals for CLU records.

Handler logic:
1. Auth check: same pattern as validation route
2. Fetch farm-budget dashboard with timeout:
   ```typescript
   let budgetData: any
   try {
     const res = await fetch('http://localhost:3001/api/dashboard', {
       signal: AbortSignal.timeout(5000),
       next: { revalidate: 0 },  // CRITICAL: no caching of cross-app calls
     })
     if (!res.ok) throw new Error(`Budget returned ${res.status}`)
     budgetData = await res.json()
   } catch {
     return NextResponse.json(
       { error: 'Farm-budget is offline — auto-populate unavailable' },
       { status: 502 }
     )
   }
   ```
   Per locked decision: no stale data fallback. Explicit error message with retry intent.

3. Fetch CLU records from Supabase:
   ```typescript
   const { data: cluRecords } = await supabase
     .from('clu_records')
     .select('id, legacy_id, farm_number, tract_number, clu, field_name, crop, fsa_acres')
     .eq('crop_year', 2026)
   ```

4. Build auto-populate proposals using enterprise-level crop matching (per research recommendation — simpler and more reliable than field-name fuzzy match):

   Create a helper function `buildAutoPopulateProposals(budgetData, cluRecords)` INSIDE the route file (not in calc.ts — this is an integration function, not pure business logic):

   a. From `budgetData.enterpriseSummaries`, build a budget crop map:
      ```typescript
      type BudgetCrop = { displayName: string; budgetAcres: number }
      const budgetCrops = new Map<string, BudgetCrop>()
      ```
      For each enterprise summary, extract crop name and total acres.

   b. Normalize crop names for matching:
      ```typescript
      function normName(s: string): string {
        return (s || '').toLowerCase().replace(/[^a-z0-9]/g, '').trim()
      }
      ```
      (Ported from fsa-acres/server.js `normName` function.)

   c. For each CLU record, find the best-matching budget crop:
      - Normalize the CLU's current crop (if any) and each budget crop
      - If a direct normalized match exists, propose that budget crop as confirmation
      - If no match and CLU has no crop, propose the budget crop with the most unmatched acres remaining
      - Track a `matchConfidence` field: 'exact' (normalized match), 'suggested' (acres-based), 'none' (no match found)

   d. Return proposals array:
      ```typescript
      interface AutoPopulateProposal {
        cluId: string
        legacyId: string
        fieldName: string | null
        farmNumber: string
        tractNumber: string
        clu: string
        fsaAcres: number
        currentCrop: string | null
        proposedCrop: string | null
        matchConfidence: 'exact' | 'suggested' | 'none'
      }
      ```

   e. Proposals should include ALL CLU records (per locked decision — not just blanks). Records with no proposed change still appear in the diff with `proposedCrop === currentCrop` and `matchConfidence: 'exact'`.

5. Return `NextResponse.json({ proposals, budgetCropCount: budgetCrops.size, cluCount: cluRecords?.length ?? 0 })`

**Anti-patterns to avoid:**
- Do NOT cache the farm-budget response (`next: { revalidate: 0 }` is mandatory)
- Do NOT call farm-registry (port 3005) for field matching — enterprise-level matching is sufficient per research
- Do NOT write any data in this endpoint — it is read-only preview. Phase 28 will build the apply endpoint.
