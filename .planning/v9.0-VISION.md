# Vision: Mobile PWA + Field Operations Logger (v9.0)

## Context

The team needs mobile access to the Glomalin platform — especially in the field where cell service is spotty. The primary gap is a way for operators to see crop plans for each field and confirm (or add) field passes as they happen. These confirmed operations feed directly into organic-cert's 3-year field histories for NOP compliance.

Today, field operations enter the system two ways:
1. **Case IH FieldOps API** — auto-synced applications (spraying, planting) from precision ag hardware
2. **Manual entry in organic-cert** — office data entry after the fact

Neither works for operations that Case IH doesn't track (rotary hoe, cultivation, mowing, flaming) or for real-time confirmation by the person actually in the field.

---

## Captured Decisions

### What the Team Sees (Read)

- **Crop plan per field** — crop, variety, planting population, planned inputs with rates
- **Planned field passes** — every operation scheduled for the field (tillage, cultivation, spraying, etc.)
- **Pass status** — which passes are confirmed vs still planned
- **Who and when** — operator name and date on every confirmed pass

### What Operators Do (Write)

- **Confirm a planned pass** — "I'm doing the planned rotary hoe pass on Kopps today" → one tap + date + operator
- **Add an unplanned pass** — "We did an extra cultivation on Simpson N" → pick field, operation type, date, operator
- **All confirmed passes push to organic-cert** — becomes part of the 3-year field history for NOP audit

### Offline is Critical

- Rural coverage gaps are real — operators must be able to confirm passes with no signal
- Queue operations locally, sync when back online
- Read-only crop plans should be available offline (cached)

### PWA Approach (Not Native)

- Progressive Web App — add-to-home-screen, service worker, offline caching
- No app store approval, no separate codebase
- Next.js 14 already supports PWA via @serwist/next (successor to next-pwa)
- Grain-tickets already has a manifest.json — extend pattern to portal

## Existing Infrastructure

### Data Sources (Already Built)

| Data | Source | API |
|------|--------|-----|
| Fields + acres | farm-registry (port 3005) | GET /api/fields |
| Crop, variety, population, inputs, rates | farm-budget (port 3001) | GET /api/fields, /api/enterprises, /api/seeds, /api/products |
| Planned field passes (budget implements) | farm-budget | field.machinery[] with passes |
| Case IH synced operations | organic-cert | FieldOps sync pipeline |
| 3-year field histories | organic-cert (Prisma/PostgreSQL) | FieldOperation, FieldEnterprise |

### Target Schema (Already Exists in organic-cert)

```prisma
model FieldOperation {
  id                String          @id
  fieldEnterpriseId String
  type              FieldOpType     // TILLAGE, PLANTING, CULTIVATION, MOWING, etc.
  operationDate     DateTime?       // null when PLANNED
  operatorId        String?         // FK to User
  acresWorked       Float?
  description       String?
  passStatus        PassStatus      // PLANNED | CONFIRMED
  plannedSource     String?         // "budget-import" | "mobile-logger"
  passNumber        Int?            // which pass (1, 2, 3)
}

enum PassStatus {
  PLANNED
  CONFIRMED
}
```

This is the exact write target. The mobile logger confirms PLANNED → CONFIRMED or creates new CONFIRMED rows.

## Architecture

### PWA Stack

- **@serwist/next** — service worker generation for Next.js 14 App Router
- **IndexedDB** (via idb) — offline operation queue + cached crop plan data
- **Background Sync API** — replay queued operations when connectivity returns
- **Web App Manifest** — add-to-home-screen with Glomalin branding

### Data Flow

```
┌─────────────────────┐     ┌──────────────┐     ┌──────────────┐
│  Mobile PWA         │────>│  Portal API  │────>│  organic-cert │
│  (offline queue)    │     │  (Next.js)   │     │  (Prisma/PG)  │
│                     │<────│              │<────│               │
│  IndexedDB cache    │     │  Supabase    │     │  FieldOps     │
└─────────────────────┘     └──────────────┘     └──────────────┘
         │                         │
         │                         v
         │                  ┌──────────────┐
         └─────────────────>│  farm-budget  │ (read-only: crop plans)
                            │  farm-registry│ (read-only: field list)
                            └──────────────┘
```

### Offline Strategy

1. **On login/sync**: Cache crop plans + field list in IndexedDB (lightweight JSON)
2. **On pass confirmation**: Write to IndexedDB queue with timestamp + operator
3. **On reconnect**: Background Sync API replays queue → Portal API → organic-cert DB
4. **Conflict resolution**: Server timestamp wins; if a pass was already confirmed (by someone else or FieldOps), skip with notification
5. **Stale data indicator**: Show "Last synced: 2h ago" badge so operators know if plans are current

### Mobile UI Approach

- **Field list** — big tap targets, search/filter, grouped by enterprise
- **Field detail** — crop info card at top, pass checklist below
- **Confirm pass** — swipe or tap checkbox → date picker (defaults today) + operator selector → done
- **Add pass** — floating action button → pick type, date, notes → queued
- **Sync indicator** — persistent badge showing queued/synced count

## Proposed Phase Structure

### Phase 44: PWA Infrastructure
- Service worker with @serwist/next, web app manifest, install prompt
- Offline shell (app loads without network), caching strategy for static assets
- IndexedDB wrapper for offline data store

### Phase 45: Crop Plan Viewer (Mobile-First)
- Supabase API route that aggregates field + enterprise + input + seed data from farm-budget + farm-registry
- Mobile-optimized field list with search, grouped by crop/enterprise
- Field detail page: crop, variety, population, planned inputs with rates, planned passes
- Offline cache: crop plan data stored in IndexedDB on each sync

### Phase 46: Field Pass Logger
- Confirm planned pass (tap → set date + operator → CONFIRMED in organic-cert)
- Add unplanned pass (pick field, op type, date, operator, notes)
- Operator selector (Supabase profiles with operator role or above)
- Write to organic-cert FieldOperation table via portal API
- `plannedSource: "mobile-logger"` tag for audit trail

### Phase 47: Offline Sync Engine
- IndexedDB queue for pending confirmations/additions
- Background Sync API integration for automatic replay
- Conflict detection (pass already confirmed by FieldOps or another user)
- Sync status UI: queued count, last sync time, error notifications
- Manual "force sync" button for impatient operators

### Phase 48: Grain Tickets PWA + Express App Offline
- Extend PWA to grain-tickets entry (already has manifest.json)
- Offline ticket entry with queue + sync (same IndexedDB pattern)
- Dashboard read-only caching for budget/FSA/insurance views

## What This Does NOT Include

- Native mobile app (iOS/Android) — PWA covers the use case
- GPS/location tracking of field passes — not needed for NOP records
- Photo attachment on passes — deferred (would need Supabase Storage)
- Push notifications — deferred (requires VAPID setup + notification permission UX)
- Equipment/implement selection on mobile — operators confirm the pass type, not which tractor
