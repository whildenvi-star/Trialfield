**FIX-03 — Partial unique index for null-label FieldEnterprise rows:**

The `@@unique([fieldId, cropYear, crop, label])` constraint in schema.prisma does NOT enforce uniqueness when `label IS NULL` because PostgreSQL treats NULL as distinct from every other NULL. Two rows with `(fieldId, 2026, "Corn", NULL)` can coexist, violating the intended constraint.

Prisma does not support partial unique indexes in schema.prisma syntax. The fix requires a raw SQL migration.

**Steps:**

1. Check if `organic-cert/prisma/migrations/` directory exists. If not, initialize Prisma migrations:
   ```bash
   cd organic-cert && npx prisma migrate dev --create-only --name init
   ```
   This creates the initial migration from the current schema. Then apply it:
   ```bash
   npx prisma migrate dev
   ```

2. Create a new migration for the partial unique index:
   ```bash
   cd organic-cert && npx prisma migrate dev --create-only --name add_partial_unique_enterprise_label_null
   ```

3. Edit the generated migration SQL file. Add the following raw SQL:
   ```sql
   -- Enforce uniqueness for FieldEnterprise rows where label IS NULL.
   -- The standard @@unique([fieldId, cropYear, crop, label]) does NOT enforce this
   -- because PostgreSQL treats NULL != NULL in unique indexes.
   CREATE UNIQUE INDEX IF NOT EXISTS "FieldEnterprise_no_label_unique"
   ON "FieldEnterprise" ("fieldId", "cropYear", "crop")
   WHERE "label" IS NULL;
   ```

4. Apply the migration:
   ```bash
   cd organic-cert && npx prisma migrate dev
   ```

5. Verify the index exists by querying pg_indexes or running Prisma Studio.

**Important:** Do NOT modify the existing `@@unique([fieldId, cropYear, crop, label])` in schema.prisma — it correctly covers the non-null label case. The partial index supplements it for the null case.

**Note:** If `prisma/migrations/` doesn't exist yet, you'll need to create the initial baseline migration first. Use `npx prisma migrate dev --create-only --name init` to baseline the current schema state, then create the partial index migration separately. Check if the PostgreSQL database already has tables — if so, mark the initial migration as already applied with `npx prisma migrate resolve --applied MIGRATION_NAME`.
