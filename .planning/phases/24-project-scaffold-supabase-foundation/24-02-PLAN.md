  <files>

  <action>
    Create `glomalin-portal/supabase/schema.sql` with the following, in order:

    1. **Role enum type:**
       ```sql
       create type user_role as enum ('admin', 'agronomist', 'operator', 'viewer');
       ```

    2. **profiles table:**
       ```sql
       create table profiles (
         id uuid primary key references auth.users(id) on delete cascade,
         role user_role not null default 'viewer',
         full_name text,
         created_at timestamptz not null default now(),
         updated_at timestamptz not null default now()
       );
       ```
       The id is a direct FK to auth.users — NOT a serial. This is the Supabase pattern where profiles.id = auth.users.id.

    3. **module_access table:**
       ```sql
       create table module_access (
         id uuid primary key default gen_random_uuid(),
         user_id uuid not null references profiles(id) on delete cascade,
         module text not null,
         granted boolean not null default false,
         created_at timestamptz not null default now(),
         updated_at timestamptz not null default now(),
         unique(user_id, module)
       );
       ```
       The module column stores module IDs from lib/modules.js (e.g., 'macro-rollup', 'farm-registry').

    4. **Enable RLS on both tables:**
       ```sql
       alter table profiles enable row level security;
       alter table module_access enable row level security;
       ```

    5. **RLS policies for profiles:**
       - `profiles_select_own`: Users can SELECT their own profile (`auth.uid() = id`)
       - `profiles_update_own`: Users can UPDATE their own profile (`auth.uid() = id`)
       - `profiles_admin_all`: Admin users can SELECT all profiles (using a subquery: `exists (select 1 from profiles where id = auth.uid() and role = 'admin')`)

    6. **RLS policies for module_access:**
       - `module_access_select_own`: Users can SELECT their own module_access rows (`auth.uid() = user_id`)
       - `module_access_admin_all`: Admin users can SELECT all module_access rows
       - `module_access_admin_manage`: Admin users can INSERT, UPDATE, DELETE all module_access rows

    7. **Auto-profile trigger function:**
       ```sql
       create or replace function handle_new_user()
       returns trigger as $$
       begin
         insert into profiles (id, role)
         values (new.id, 'viewer');
         return new;
       end;
       $$ language plpgsql security definer;
       ```

    8. **Trigger binding:**
       ```sql
       create trigger on_auth_user_created
         after insert on auth.users
         for each row execute function handle_new_user();
       ```

    9. **updated_at trigger** for both tables:
       Create a generic `set_updated_at()` function and bind it to profiles and module_access.

    Add comments at the top of the file:
    ```sql
    -- Glomalin Portal: Supabase Schema
    -- Apply via Supabase Dashboard > SQL Editor
    -- After applying, run seed.sql for test data
    ```

    IMPORTANT: Use `auth.uid()` (NOT `current_user` or session variables) for RLS — this is the Supabase pattern.
    IMPORTANT: The trigger function must be `security definer` so it can insert into profiles even though the new user doesn't have a profile yet.
  </action>
