**1. Create Supabase tables via the migration script.**

The migration script at `glomalin-portal/scripts/migrate-fsa.ts` should handle BOTH schema creation (via raw SQL) and data upsert. Use `@supabase/supabase-js` with `SUPABASE_SERVICE_ROLE_KEY` to bypass RLS.

**Schema (5 tables) ŌĆö execute via `supabase.rpc()` or raw SQL:**

Create tables per the schema in 27-RESEARCH.md:
- `clu_records`: UUID PK, `legacy_id text unique not null`, `crop_year integer not null default 2026`, all fields from research schema. `grain_plant_date` as TEXT (not date ŌĆö source has mixed formats like '2010', 'TBD', ''). Index on `crop_year` and `farm_number`.
- `insurance_pricing`: UUID PK, `legacy_id text unique not null`, `crop text`, `year integer`, `spring_price numeric(10,4)`, `fall_price numeric(10,4)`, unique index on `(crop, year)`.
- `insurance_policies`: UUID PK, `legacy_id text unique not null`, policy-only fields (no claim fields). Fields: `farm_name`, `farm_number`, `line_number`, `policy_number`, `crop`, `policy_year integer default 2026`, `planted_acres`, `fsa_acres_manual`, `guarantee`, `actual`, `coverage_level integer default 75`, `unit_type`, `premium_per_acre`, `agent_name`, `prevented_planting boolean default false`, `prevented_planting_acres`, `notes`.
- `claims`: UUID PK, `policy_id uuid not null references insurance_policies(id) on delete cascade`, `claim_status text default 'none'`, `claim_number`, `loss_type`, `adjuster_name`, `adjuster_phone`, `claim_filed_date date`, `claim_paid_date date`, `claim_paid_amount numeric(10,2)`.
- `gcs_enrollments`: UUID PK, `legacy_id text unique not null`, fields per research schema. Index on `(farm_number, tract_number)`.

All tables get `created_at timestamptz default now()`. Tables with mutable data get `updated_at timestamptz default now()`.

Enable RLS on all 5 tables. Create a simple `authenticated_read` SELECT policy on each: `for select using (auth.role() = 'authenticated')`.

Use `CREATE TABLE IF NOT EXISTS` so the script is re-runnable without error if tables already exist.

**2. Migrate data from fsa-acres/data/data.json.**

Read `../../fsa-acres/data/data.json` using `fs.readFileSync`. The file structure is:
```
{ cluRecords: [...], pricing: [...], insurancePolicies: [...], gcsEnrollments: [...], settings: { year: 2026 } }
```

Map and upsert each table:

- **CLU records (444 rows):** Map camelCase ŌåÆ snake_case. Use `upsert` with `onConflict: 'legacy_id'`. Batch in chunks of 500.
- **Pricing (22 rows):** Map `id` ŌåÆ `legacy_id`, `springPrice` ŌåÆ `spring_price`, etc. Upsert on `legacy_id`.
- **Insurance policies (3 rows):** Map ONLY policy fields (not claim fields). Use `legacy_id` upsert. For `ins_482` (suspect data ŌĆö no farm, no crop, actual=40000), insert with `notes: 'VERIFY ŌĆö data may be corrupt (no farm/crop, actual=40000)'`.
- **Claims (3 rows ŌĆö one per insurance policy):** After upserting insurance_policies, query back the inserted rows to get their UUIDs. For each policy that has claim data (`claimStatus`, `claimNumber`, etc.), insert a claims row with `policy_id` set to the UUID. `ins_478` has `claimStatus: 'none'` ŌĆö still insert a row for FK chain completeness.
- **GCS enrollments (149 rows):** Map fields, upsert on `legacy_id`.

**2. Create the CLU records read API.**

Create `glomalin-portal/src/app/api/fsa/clu-records/route.ts` with a GET handler.

Follow the existing Route Handler pattern from `api/admin/users/route.ts`:
- Import `createClient` from `@/lib/supabase/server`
- Import `NextResponse` from `next/server`
- Auth check: `supabase.auth.getUser()` ŌĆö return 401 if no user
- Read `year` from URL searchParams (default to 2026 if not provided)
- Query: `supabase.from('clu_records').select('*').eq('crop_year', year).order('farm_number').order('tract_number').order('clu')`
- Return `NextResponse.json({ records: data, count: data?.length ?? 0 })`
- Handle Supabase errors with 500 response

The endpoint accepts `?year=2026` query parameter. The ordering (farm_number, tract_number, clu) groups records logically for the Phase 28 card UI.

**Note:** Module access is enforced by the middleware for the page route (`/app/fsa-578`). The API route at `/api/fsa/clu-records` is NOT under `/app/` so middleware does not enforce module access on it ŌĆö the auth check (getUser) is sufficient since only authenticated users call it. If tighter access control is needed, it can be added in Phase 28.
