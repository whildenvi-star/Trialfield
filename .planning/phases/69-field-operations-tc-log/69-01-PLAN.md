Create `glomalin-portal/src/app/api/field-ops/tcs/route.ts` with two handlers.

**Auth pattern:** Use Supabase server-side auth (createServerClient from @supabase/ssr) to get the current session — NOT getMobileUser (which requires Bearer token). All portal pages use cookie-based SSR auth. See existing portal API routes in `/api/insurance/` or `/api/marketing/` for the correct SSR pattern. Return 401 if no session.

**GET handler — list TCs for a field/year:**

Query params: `fieldId` (required), `year` (optional, defaults to current year as number).

Steps:
1. Validate fieldId present, return 400 if missing.
2. Use `resolveFieldEnterpriseId(fieldId)` from `@/app/api/mobile/_lib/cert-bridge` to get certFieldEnterpriseId. Wrap in try/catch — if field has no organic-cert enterprise for that year, return `{ tcs: [], noEnterprise: true }` (not an error — field may be conventional).
3. Fetch `GET /api/field-enterprises/{certFieldEnterpriseId}` from organic-cert via `fetchCertService` (from `@/app/api/mobile/_lib/proxy`). This returns the FieldEnterprise detail with `fieldOperations` array (each op has id, type, status, operationDate, plannedSource, notes, and an operator join).
4. Filter `fieldOperations` where `plannedSource === "field-ops-tc"` AND operationDate year matches the requested year.
5. Return `{ tcs: TcRecord[], year, fieldId }` where TcRecord has: `{ id, operationType, operationDate, notes, tcByName, tcByCertUserId, createdAt }`. Map `op.operator?.name` → `tcByName`, `op.operator?.id` → `tcByCertUserId`.

**POST handler — create a TC:**

Body: `{ fieldId: string, operationType: string, operationDate: string (ISO date), notes?: string, tcByOverrideCertUserId?: string }`

Validation: all three required fields must be present. `operationType` must be one of: Tillage, No-Till, Planting, Herbicide, Fertilizer, Scouting, Harvest, Hauling, Other.

Steps:
1. Get caller's `cert_user_id` from Supabase profile. Return 403 if null.
2. Map operationType to organic-cert FieldOpType using `OP_TYPE_MAP` from cert-bridge. Add `'No-Till': 'TILLAGE'` and `'Hauling': 'OTHER'` to the mapping usage (do NOT modify cert-bridge.ts itself — map locally with a spread: `const FULL_MAP = { ...OP_TYPE_MAP, 'No-Till': 'TILLAGE', Hauling: 'OTHER' }`).
3. Resolve `certFieldEnterpriseId` via `resolveFieldEnterpriseId(fieldId)`. Return 422 with `{ error: 'No organic-cert enterprise for this field/year' }` if throws.
4. POST to `fetchCertService('/api/field-enterprises/{certFieldEnterpriseId}/operations')` with body:
   ```json
   {
     "type": "<mapped FieldOpType>",
     "status": "CONFIRMED",
     "operationDate": "<operationDate>",
     "notes": "<notes or empty string>",
     "plannedSource": "field-ops-tc",
     "dataSource": "MANUAL",
     "operatorId": "<tcByOverrideCertUserId ?? callerCertUserId>"
   }
   ```
5. Return cert response as-is if 2xx, else return `{ error: 'Failed to create TC', status }` with same status code.

TypeScript: no `any` without eslint-disable comment. Use explicit types for body and response shapes.

**File 1: `glomalin-portal/src/app/api/field-ops/tcs/[id]/route.ts`**

DELETE handler for a single TC.

Params: `id` (the organic-cert FieldOperation ID from the URL segment).
Query params: `fieldEnterpriseId` (required — needed to construct the cert URL), `tcByCertUserId` (the cert user who created the TC, for ownership check).

Auth: Same SSR cookie auth pattern. Get caller's Supabase profile role and cert_user_id.

Permission logic:
- Admin role → always allowed to delete.
- Any other role → only allowed if caller's `cert_user_id === tcByCertUserId` query param (ownership check).
- If not allowed, return 403 `{ error: 'You can only delete TCs you created' }`.

Steps:
1. Validate `fieldEnterpriseId` and `id` present. Return 400 if missing.
2. Apply permission check as above.
3. DELETE `fetchCertService('/api/field-enterprises/{fieldEnterpriseId}/operations/{id}')` — check that organic-cert's operations endpoint supports DELETE (it does from Phase 46 pattern; if it returns 405, fall back to POST with `_method=DELETE` — but do not add that fallback unless the DELETE call actually fails in testing).
4. If cert returns 404, return 404. If cert returns 2xx, return `{ success: true }`. Otherwise return the cert error status.

**File 2: `glomalin-portal/src/app/api/field-ops/operators/route.ts`**

GET handler returning all profiles with operator/agronomist/admin roles.

This is essentially identical to `/api/mobile/operators/route.ts` but:
- Uses SSR cookie auth instead of Bearer token auth.
- Does NOT require `cert_user_id IS NOT NULL` — the TC log allows selecting any admin/operator/agronomist regardless of whether they have a cert account (since the portal stores tcByName as a display string).
- Returns `{ operators: [{ id, fullName, role }] }` (no certUserId needed — the portal will store the name, not the cert ID).

Implementation: Use `supabaseAdmin` (service role) to query profiles table. Filter `role IN ('operator', 'agronomist', 'admin')`. Order by `full_name ASC`. Map to `{ id, fullName: full_name ?? id, role }`.

TypeScript: no errors. Use createClient with service role key for the admin client.
