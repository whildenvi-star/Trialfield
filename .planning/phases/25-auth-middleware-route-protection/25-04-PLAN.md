**Schema migration** — `supabase/migrations/001-admin-write-policies.sql`:
The existing schema has `profiles_admin_all` as SELECT-only for admins. Add:
1. Admin update policy on profiles: `create policy profiles_admin_update on profiles for update using (exists (select 1 from profiles where id = auth.uid() and role = 'admin'));`
2. This allows admins to update any user's role. The `module_access_admin_manage` policy already covers ALL operations on module_access.

Include a comment at top: "Run in Supabase SQL Editor after schema.sql"

**GET /api/admin/users** — `api/admin/users/route.ts`:
- Create server Supabase client via `createClient()` from `@/lib/supabase/server`
- Verify caller is admin: `supabase.auth.getUser()` then `supabase.from('profiles').select('role').eq('id', user.id).single()` — if not admin, return 403
- Fetch all profiles: `supabase.from('profiles').select('id, full_name, role, created_at, updated_at')`
- Fetch all module_access: `supabase.from('module_access').select('user_id, module, granted')`
- Also fetch user emails: Since profiles doesn't store email (it's in auth.users), use Supabase Admin client for this. Create a separate admin client using `createClient` from `@supabase/supabase-js` with `SUPABASE_SERVICE_ROLE_KEY` env var and `auth: { autoRefreshToken: false, persistSession: false }`. Call `supabase.auth.admin.listUsers()` to get emails.
- Merge profiles + emails + module_access into a combined response: `{ users: [{ id, email, fullName, role, lastSignIn, modules: { 'macro-rollup': true, ... } }] }`
- Return JSON response

**PATCH /api/admin/users/[id]/role** — `api/admin/users/[id]/role/route.ts`:
- Verify caller is admin (same pattern as GET)
- Read body: `{ role: 'admin' | 'agronomist' | 'operator' | 'viewer' }`
- Prevent self-role-change: if `params.id === currentUser.id`, return 403 with error "Cannot change your own role"
- Update: `supabase.from('profiles').update({ role }).eq('id', params.id)`
- Return 200 with updated profile

**PATCH /api/admin/users/[id]/access** — `api/admin/users/[id]/access/route.ts`:
- Verify caller is admin
- Read body: `{ module: string, granted: boolean }`
- Upsert: `supabase.from('module_access').upsert({ user_id: params.id, module, granted }, { onConflict: 'user_id,module' })`
- Return 200 with updated access record

**POST /api/admin/invite** — `api/admin/invite/route.ts`:
- Verify caller is admin
- Read body: `{ email: string, role: 'admin' | 'agronomist' | 'operator' | 'viewer' }`
- Use Supabase Admin client (service_role key): `adminClient.auth.admin.inviteUserByEmail(email)`
- After invite succeeds, the auto-profile trigger creates a viewer profile. If role !== 'viewer', update the profile: `supabase.from('profiles').update({ role }).eq('id', newUser.id)`
- Return 201 with `{ id, email }`

**Important for all API routes:**
- Use `NextResponse.json()` for responses
- Use `NextRequest` type for request parameter
- Extract route params via the second argument: `{ params }: { params: { id: string } }`
- Add `SUPABASE_SERVICE_ROLE_KEY` to `.env.local.example` with comment about where to find it

**User table:**
- Styled with dark soil: `border border-soil-border`, header row `bg-soil-surface`
- Columns: Name, Email, Role, Modules (5 toggle columns), Last Login
- Data fetched on mount via `fetch('/api/admin/users')` — use useEffect + useState
- Each row: `border-b border-soil-border hover:bg-soil-surface/50`

**Role dropdown (per row):**
- `<select>` element with soil styling: `bg-soil-bg border border-soil-border text-soil-text rounded px-2 py-1`
- Options: admin, agronomist, operator, viewer
- Current role pre-selected

- If this is the current admin's own row: `disabled` attribute on the select (per CONTEXT.md "admin cannot edit their own role")
- While saving: brief loading indicator (opacity reduction or spinner)

**Module toggle switches (per row, per module):**
- One toggle per module from MODULES array (5 toggles total)
- Use a styled checkbox or button that looks like a toggle switch:
  - ON state: `bg-soil-accent` with a circle/indicator on the right
  - OFF state: `bg-soil-border` with indicator on the left
  - Implement with a `<button>` + CSS (div with rounded-full, transition)
- On click: immediately PATCH `/api/admin/users/{id}/access` with `{ module: moduleId, granted: !currentValue }`
- While saving: brief opacity reduction

**Last login column:**
- Display `lastSignIn` from API as relative time or formatted date
- If null: "Never" in soil-muted

**State management:**
- `users` state: array of user objects from API
- `loading` state: boolean for initial load
- `savingCell` state: `{ userId: string, field: string } | null` to track which cell is saving (for loading indicators)
- After any PATCH succeeds, update the local `users` state directly (optimistic-ish — update after success response) rather than re-fetching the entire list

**Styling notes:**

- Table is responsive: horizontal scroll on small screens (`overflow-x-auto`)
- Module column headers: abbreviated module labels or use the `label` from MODULES
