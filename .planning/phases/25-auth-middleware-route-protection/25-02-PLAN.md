Create `glomalin-portal/src/lib/supabase/middleware.ts` — a Supabase client factory specifically for Next.js middleware.

Middleware cannot use `cookies()` from `next/headers` — it receives the request object and must forward cookies via `NextRequest`/`NextResponse`.

Export a function `createClient(request: NextRequest)` that:
1. Creates a `NextResponse.next()` response (with `request: { headers: request.headers }`)
2. Uses `createServerClient` from `@supabase/ssr` with cookie handlers that:
   - `getAll()`: reads from `request.cookies.getAll()`
   - `setAll(cookiesToSet)`: sets cookies on BOTH the request AND the response (important for session refresh — the request cookies are forwarded to Server Components, the response cookies are sent to the browser)
3. Returns `{ supabase, response }` — the caller uses `supabase` for auth checks and returns `response` (which carries refreshed session cookies)

This follows the standard Supabase + Next.js middleware pattern from the @supabase/ssr docs.

<task type="auto">
  <name>Task 2: Create Next.js middleware with auth + RBAC route protection</name>
  <files>glomalin-portal/src/middleware.ts</files>
  <action>
Create `glomalin-portal/src/middleware.ts` with the following logic:

**Route classification (define at top):**
- Public routes: `/`, `/login`, `/forgot-password`, `/_next`, `/favicon.ico`, `/api/auth` — NO auth check
- Admin routes: paths starting with `/admin`
- Module routes: paths starting with `/app/` followed by a module slug
- Protected routes: everything else (e.g., `/dashboard`)

**Middleware function flow:**

```
1. Create Supabase client via createClient(request) from lib/supabase/middleware
2. Call supabase.auth.getUser() to refresh session and get current user
3. If public route → return response (pass through)
4. If no user (unauthenticated):
   - If session cookie existed but getUser failed → redirect to /login?expired=true
   - Otherwise → redirect to /login
5. If admin route (/admin*):
   - Query profiles table: supabase.from('profiles').select('role').eq('id', user.id).single()
   - If role !== 'admin' → redirect to /dashboard (SILENT — no query param, no hint per CONTEXT.md)
   - If admin → pass through
6. If module route (/app/{moduleId}*):
   - Extract moduleId from pathname (second segment after /app/)
   - Query module_access: supabase.from('module_access').select('granted').eq('user_id', user.id).eq('module', moduleId).single()
   - If no record or granted === false → redirect to /dashboard?denied={moduleId}
   - If granted → pass through
7. All other protected routes → pass through (user is authenticated)
8. Always return the response object from createClient (carries refreshed cookies)
```

**Matcher config** — export a `config` object:
```typescript
export const config = {
  matcher: [
    '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)',
  ],
}
```

This matches all routes except static assets. The public route check happens inside the middleware function itself.

**Important implementation notes:**
- Use `NextResponse.redirect(new URL('/login', request.url))` for redirects
- The `createClient` from `lib/supabase/middleware` returns `{ supabase, response }` — always return `response` for non-redirect cases (it carries refreshed session cookies)
- For checking if session expired vs never existed: check if `request.cookies.getAll().some(c => c.name.includes('sb-'))` — if Supabase cookies exist but getUser() returned no user, the session expired
- RBAC-01 (role defaulting to viewer) is already handled by the DB trigger from Phase 24 — middleware just reads the role
