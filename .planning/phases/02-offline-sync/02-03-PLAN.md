**CRITICAL constraint from RESEARCH.md open question 2:**
The current `processQueue` already skips "already confirmed" 409s silently. This is CORRECT behavior for pass confirmations — do NOT break it. The new conflict logic must only trigger when:
- The 409 response body contains a JSON payload with a `serverVersion` or `serverPayload` field (or equivalent indicator of a competing write worth showing the user)
- Distinguished from a "already confirmed" 409 by presence/absence of a parseable server version in the body

**Changes to make:**

1. **Extend SyncResult interface:**
```typescript
export interface SyncResult {
  synced: number
  skipped: SyncSkip[]
  failed: SyncFailure[]
  conflicts: ConflictRecord[]   // ADD THIS — import ConflictRecord from '@/lib/offline/types'
  total: number
}
```
Initialize `conflicts: []` at the start of processQueue alongside existing `synced`, `skipped`, `failed` initializations.

2. **Conflict detection in 409 handling:**
In the existing 409 response branch, after the current "already confirmed" skip check:
```typescript
// If 409 body contains a serverPayload key, it's a true data conflict
const body = await response.json().catch(() => null)
if (body && (body.serverPayload || body.serverVersion)) {
  const conflict: ConflictRecord = {
    id: crypto.randomUUID(),
    type: operation.type as ConflictRecord['type'],
    fieldId: operation.fieldId ?? '',
    operationDate: operation.operationDate ?? new Date().toISOString(),
    localPayload: operation.payload as Record<string, unknown>,
    serverPayload: body.serverPayload ?? body.serverVersion ?? body,
    createdAt: new Date().toISOString(),
    resolved: 0
  }
  // Write to IDB conflicts store
  const db = await getDb()
  await db.put('conflicts', conflict)
  result.conflicts.push(conflict)
  // Mark operation as skipped (not failed) so it doesn't retry
  result.skipped.push({ id: operation.id, reason: 'conflict' })
  await offlineQueue.markSynced(operation.id)   // remove from queue
} else {
  // existing "already confirmed" skip behavior — unchanged
}
```

3. **Ensure `result` is returned with `conflicts` field.** The function already returns `result` — just confirm `conflicts` is on the object.

4. **Check field names:** Read the file first to confirm the correct field names on operation objects (`operation.type`, `operation.fieldId`, etc.) and use the actual field names present in the type. Check `QueuedOperation` in types.ts for canonical field names.

Do NOT modify the retry logic, backoff, auth refresh, or any other existing behavior.

**conflict-drawer.tsx:**
Create `src/components/offline/conflict-drawer.tsx` as a `'use client'` component.

The drawer must stay under 100 lines of JSX (per RESEARCH.md Pitfall 4 — conflict scope creep warning). Keep it strictly to: list of unresolved conflicts, two-button resolution per conflict, nothing else.

Behavior:
1. **On mount:** Listen for `window` CustomEvent `'sync:conflicts'` (dispatched by useSyncStatus in Plan 01 Task 2). When received, set `open` state to true.
2. **On open:** Read all unresolved conflicts from IDB: `const db = await getDb(); const all = await db.getAllFromIndex('conflicts', 'by-resolved', 0)`
3. **Display:** For each conflict, show:
   - Type label: `'confirm-pass'` → "Confirm Pass", `'add-pass'` → "Add Pass", `'observation'` → "Field Observation"
   - Field ID (if present), operation date (formatted as locale date string)
   - Local value section: "Your version" header + JSON.stringify(localPayload, null, 2) in a `<pre>` block (capped height, overflow scroll)
   - Server value section: "Server version" header + JSON.stringify(serverPayload, null, 2) in same styled `<pre>` block
   - Two buttons: "Keep mine" and "Keep server version"
4. **On resolution:**
   - "Keep mine": mark `resolved: 1` in IDB (`await db.put('conflicts', { ...conflict, resolved: 1 })`), remove from displayed list
   - "Keep server version": same IDB mark as resolved, remove from displayed list
   - Note: the actual data reconciliation API call (sending the chosen version back to the server) is out of scope for Phase 2 — just record the resolution locally. Field observations rarely conflict and this is a safety net, not a merge engine.
5. **When all conflicts resolved:** `open` state auto-sets to false (or user can close with an X button)
6. **Unresolved conflicts persist** across sessions — drawer only opens on new `sync:conflicts` event, not on page load. Users can ignore conflicts.

Use the same sheet/dialog implementation as `queue-detail-sheet.tsx` for the drawer container.
