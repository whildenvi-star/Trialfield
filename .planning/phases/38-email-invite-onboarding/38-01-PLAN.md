1. In `invite/route.ts`: Add `redirectTo` option to `inviteUserByEmail` call so the invite email links to the production domain's callback.

   Build the redirect URL using `NEXT_PUBLIC_SITE_URL` env var (already documented in .env.example), falling back to the request origin via `request.headers.get('origin')`, then finally `http://localhost:3000`. The redirectTo should point to `/auth/callback`.

   Change line 46-48 from:
   ```
   const { data: inviteData, error: inviteError } = await adminClient.auth.admin.inviteUserByEmail(body.email)
   ```
   to pass `{ redirectTo: siteUrl + '/auth/callback' }` as the second argument.

   Also add optional `modules` array to the invite body (string[] of module IDs). After creating the user, if `body.modules` is provided and non-empty, insert module_access rows for each module with `granted: true` using the admin supabase client (NOT the per-user client, since RLS may block inserts for the caller on behalf of the new user). This lets admin grant module access at invite time instead of requiring a separate step.

2. In `callback/route.ts`: The current code checks `session?.user?.recovery_sent_at` to detect password recovery. For invited users, Supabase uses `type=invite` in the URL hash/params. However, with PKCE flow (which Next.js SSR uses), both invite and recovery tokens arrive as `code` params and exchange for a session.

   After `exchangeCodeForSession`, check two conditions to redirect to `/reset-password`:
   - The user has no `last_sign_in_at` or it was just set (first login = invite acceptance)
   - OR `recovery_sent_at` is recent (password reset flow)

   Simplify: after code exchange, check if the URL has a `type` query param. Supabase appends `type=invite` or `type=recovery` to the redirect URL. If `type` is `invite` or `recovery`, redirect to `/reset-password`. Otherwise redirect to `next` param (default `/dashboard`).

   Read `searchParams.get('type')` alongside `code` and `next`. If type is `invite` or `recovery`, redirect to `/reset-password` after successful code exchange. Remove the session/recovery_sent_at check (fragile, depends on timing).

1. In `auth.ts`: The `resetPassword` action currently builds redirectTo from `headers().get('origin')`. In production behind Caddy, the origin header may be the internal hostname. Replace the origin detection with:
   ```
   const siteUrl = process.env.NEXT_PUBLIC_SITE_URL || headersList.get('origin') || headersList.get('host') || 'http://localhost:3000'
   const redirectTo = siteUrl.startsWith('http') ? `${siteUrl}/auth/callback` : `https://${siteUrl}/auth/callback`
   ```
   This ensures password reset emails link to the production domain when NEXT_PUBLIC_SITE_URL is set.

2. In `admin/page.tsx`: Add module access checkboxes to the invite form so admin can grant modules at invite time (instead of inviting then separately toggling modules).

   Add a state variable `inviteModules` as `Record<string, boolean>` initialized to all false. Below the role select in the invite form, add a row of small checkboxes for each module from the MODULES import. When submitting, include `modules: Object.entries(inviteModules).filter(([_, v]) => v).map(([k]) => k)` in the POST body.

   Style the module checkboxes as a compact inline row with small labels, matching the existing glomalin-* design tokens. Use the same font-mono text-xs style as the rest of the form.
