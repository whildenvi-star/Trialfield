**Part A: Cross-group drag-and-drop**

Add a `makeFoGroupsDraggable()` function that is called at the end of `renderFieldOpsPanel()` after setting `#fo-groups-container` innerHTML.

The function must support:
1. **Within-group reorder** — same as existing `makeRowsSortable()` logic but scoped to rows within the same `.fo-group`
2. **Cross-group move** — allow dropping a row from one group onto a different group's tbody

Implementation approach:

```js
function makeFoGroupsDraggable() {
  var dragSrc = null; // { itemType, sourceIdx, origGroup }

  document.querySelectorAll('.fo-table tbody tr[data-drag-idx]').forEach(function(row) {
    var handle = row.querySelector('.drag-handle');
    if (!handle) return; // seed rows have no handle, skip

    handle.addEventListener('mousedown', function() { row.draggable = true; });
    handle.addEventListener('mouseup',   function() { row.draggable = false; });

    row.addEventListener('dragstart', function(e) {
      dragSrc = {
        itemType: row.getAttribute('data-item-type'),
        sourceIdx: parseInt(row.getAttribute('data-source-idx')),
        origGroup: row.closest('.fo-group').getAttribute('data-group')
      };
      e.dataTransfer.effectAllowed = 'move';
      row.classList.add('drag-src');
    });

    row.addEventListener('dragend', function() {
      row.draggable = false;
      row.classList.remove('drag-src');
      document.querySelectorAll('.fo-group').forEach(function(g) {
        g.classList.remove('fo-drop-target');
      });
      dragSrc = null;
    });
  });

  // Group-level drop zone (for cross-group drops — drop onto group header or group body)
  document.querySelectorAll('.fo-group').forEach(function(group) {
    var targetGroup = group.getAttribute('data-group');

    group.addEventListener('dragover', function(e) {
      if (!dragSrc) return;
      e.preventDefault();
      e.dataTransfer.dropEffect = 'move';
      if (dragSrc.origGroup !== targetGroup) {
        group.classList.add('fo-drop-target');
      }
    });

    group.addEventListener('dragleave', function(e) {
      if (!e.relatedTarget || !group.contains(e.relatedTarget)) {
        group.classList.remove('fo-drop-target');
      }
    });

    group.addEventListener('drop', function(e) {
      e.preventDefault();
      group.classList.remove('fo-drop-target');
      if (!dragSrc || dragSrc.origGroup === targetGroup) return; // same group = within-group handled by row dragover

      // Cross-group: move item and set operationGroup override
      var arr = dragSrc.itemType === 'pass'
        ? currentField.machinery
        : currentField.inputs;
      if (!arr || dragSrc.sourceIdx >= arr.length) return;
      arr[dragSrc.sourceIdx].operationGroup = targetGroup; // override classifier
      renderFieldOpsPanel();
      updatePreview();
    });
  });
}
```

**operationGroup override in classifyItem**: In `renderFieldOpsPanel()`, when classifying each item, check `item.operationGroup` first — if set, use it directly instead of calling `FieldOpsGroups.classifyItem()`. This persists cross-group moves across re-renders and saves with the field data.

Add CSS for `.fo-drop-target` in style.css (can be done inline in the JS as a `<style>` injection or by adding to the style.css additions from Plan 02 — add to style.css):
```css
.fo-group.fo-drop-target {
  outline: 2px dashed var(--primary);
  outline-offset: -2px;
}
```

**Part B: Inline add-item form**

Replace the no-op `.fo-add-item` click handler in `renderFieldOpsPanel()` with an actual inline add form.

When `.fo-add-item` is clicked for a group, inject an inline form row at the bottom of that group's `<tbody>`:

```html
<tr class="fo-add-row" data-adding-group="{groupName}">
  <td></td>
  <td colspan="5">
    <div style="display:flex;gap:0.5rem;align-items:center;flex-wrap:wrap;padding:0.25rem 0">
      <select class="fo-add-type" style="font-size:0.75rem;width:90px">
        <option value="input">Input</option>
        <option value="pass">Pass</option>
      </select>
      <input type="text" class="fo-add-name" placeholder="Product or implement name..."
        style="font-size:0.75rem;flex:1;min-width:140px" list="prod-search-list">
      <input type="number" class="fo-add-qty" placeholder="Qty/Passes" step="0.1" min="0"
        style="font-size:0.75rem;width:75px">
      <button class="btn-sm btn-primary fo-add-confirm" style="font-size:0.72rem;padding:0.2rem 0.5rem">Add</button>
      <button class="btn-sm fo-add-cancel" style="font-size:0.72rem;padding:0.2rem 0.5rem">✕</button>
    </div>
  </td>
  <td></td>
</tr>
```

- Use the existing `list="prod-search-list"` datalist for product name hints (same as used in product autocomplete, already rendered in the editor DOM)
- Add an `impl-search-list` datalist hint for the implement field — toggle the `list` attribute based on the selected type

On **Confirm**:
- If type = 'input': push `{ id: util.generateId('inp'), productName: name, quantity: qty, season: 'Spring', operationGroup: groupName }` to `currentField.inputs`
- If type = 'pass': push `{ id: util.generateId('mach'), implementName: name, passes: qty || 1, operationGroup: groupName }` to `currentField.machinery`
- Call `renderFieldOpsPanel()` and `updatePreview()`

On **Cancel**: remove the `.fo-add-row` and restore the `+ Add to {Group}` button.

Only one add-row open at a time: if another group's `+ Add` button is clicked while one is open, close the open one first.

The `operationGroup` field is set so the new item appears in the correct group immediately and persists across re-renders (same override mechanism as cross-group DnD).

**datalist for implements**: Add a `<datalist id="impl-search-list">` with options from `window.refData.implements` to the Field Ops panel HTML generation inside `renderFieldOpsPanel()` — insert it as a hidden element appended to `#ed-fieldops-unified-body` after the groups container render.
