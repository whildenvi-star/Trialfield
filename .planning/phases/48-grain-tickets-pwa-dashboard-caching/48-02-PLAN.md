2. Add a list of dashboard API URL patterns to cache:
   - Requests to Express app dashboard/summary endpoints accessed via portal iframes:
     - Farm-budget summary: URLs matching `/api/dashboard` or `/api/forecast` on port 3001
     - FSA acreage summary: URLs matching `/api/summary` or `/api/fields` on port 3002
     - Insurance summary: URLs matching `/api/insurance/policies` on the portal itself
   - Since these are cross-origin iframe embeds, the caching must handle the embedded app responses
   - Pattern: match URLs containing these path segments regardless of origin/port

3. Implement stale-while-revalidate strategy for matched dashboard URLs:
   - On fetch: check `dashboard-cache` for existing response
   - If cached: return cached response immediately, ALSO fire network request in background
   - On successful network response: update cache with new response, store timestamp in a companion cache entry `{url}__timestamp` with value `{ cachedAt: Date.now() }`
   - If not cached AND network fails: return offline fallback response `{ offline: true, error: 'No cached data available' }` with 200 status
   - If cached AND network fails: serve stale cache (already returned above)

4. Add cache cleanup in activate event:
   - Delete `dashboard-cache` entries older than 48 hours (generous buffer beyond 24h staleness threshold)

5. Add message handler for `{ type: 'clear-dashboard-cache' }`:
   - Deletes all entries from `dashboard-cache`
   - Posts back `{ type: 'dashboard-cache-cleared' }`

Note: The portal sw.js uses vanilla JS (not TypeScript). Follow the existing code style with `var` declarations and `function` expressions (matching the grain-tickets sw.js pattern and existing portal sw.js patterns).
