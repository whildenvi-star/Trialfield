Wire CropLot creation into the staged-ops approve flow for synced harvest operations, with bu-to-lbs yield conversion at the approval boundary.

Purpose: Synced harvest records from Case IH currently get a HarvestEvent but no CropLot — making them invisible to the harvest log and mass balance in PDF reports. This plan closes that gap by creating/updating CropLots atomically alongside HarvestEvents during approval.

Output: A yield converter utility (`yield-converter.ts`) and a refactored approve handler that wraps HarvestEvent + CropLot writes in a single Prisma `$transaction`.
</objective>

**FieldEnterprise year matching improvement** (per research pitfall 3):
Before the `fieldEnterprise` lookup (current line 135), when `stagedOp.operationDate` is not null, extract the year and prefer a FieldEnterprise where `cropYear === operationDate.getFullYear()`. Fall back to the current behavior (latest by cropYear desc) only if no year-matched enterprise exists. Implementation:

```typescript
let fieldEnterprise;
if (stagedOp.operationDate) {
  const opYear = stagedOp.operationDate.getFullYear();
  fieldEnterprise = await prisma.fieldEnterprise.findFirst({
    where: { fieldId: stagedOp.mappedFieldId, cropYear: opYear },
  });
}
if (!fieldEnterprise) {
  fieldEnterprise = await prisma.fieldEnterprise.findFirst({
    where: { fieldId: stagedOp.mappedFieldId },
    orderBy: { cropYear: "desc" },
  });
}
```

**Replace the harvest branch** (from `if (isHarvestType) {` through the return statement) with a `$transaction` block:

1. Inside `prisma.$transaction(async (tx) => { ... })`:
   - Convert yield to lbs using `convertYieldToLbs(productsData?.yield, productsData?.yieldUnit, acresHarvested, fieldEnterprise.crop)` — store as `netWeightLbs`
   - Create `HarvestEvent` via `tx.harvestEvent.create(...)` — same data as current, but add `netWeight: netWeightLbs`
   - Look up existing CropLot: `tx.cropLot.findFirst({ where: { fieldEnterpriseId } })`
   - If exists: `tx.cropLot.update({ where: { id: existingLot.id }, data: { quantityLbs: { increment: netWeightLbs ?? 0 } } })` — do NOT overwrite `harvestEventId` (per user decision: accumulate into existing lot)
   - If not exists: Look up Field name via `tx.field.findUnique({ where: { id: fieldEnterprise.fieldId } })`, call `generateLotNumber(fieldEnterprise.cropYear, fieldEnterprise.crop, field.name)`, then `tx.cropLot.create(...)` with `harvestEventId: harvestEvent.id`, `lotNumber`, `crop: fieldEnterprise.crop`, `organicStatus: fieldEnterprise.organicStatus`, `quantityLbs: netWeightLbs ?? 0`
   - Mark SyncedOperation APPROVED via `tx.syncedOperation.update(...)` — same data as current but INSIDE the transaction
   - Return `{ harvestEvent, cropLot, lotIsNew: boolean }`

2. After the transaction, log the audit event (audit logging can stay outside the transaction — it's non-critical)

3. Return JSON response: `{ success: true, domainRecordType: "HarvestEvent", domainRecordId: result.harvestEvent.id, cropLot: { id: result.cropLot.id, lotNumber: result.cropLot.lotNumber, isNew: result.lotIsNew } }`

**CRITICAL — anti-patterns to avoid:**
- Do NOT use `prisma.*` inside the `$transaction` callback — use `tx.*` for every DB call (reads and writes)
- Do NOT use `prisma.cropLot.upsert` with `lotNumber` as key — `lotNumber` is globally unique but not per-enterprise unique. Use `findFirst({ where: { fieldEnterpriseId } })` then branch
- Do NOT apply the manual-path lot suffix pattern (-2, -3) — synced harvests accumulate into one CropLot per enterprise (user locked decision)
- Do NOT overwrite `harvestEventId` when updating an existing CropLot

**Non-harvest branch:** Leave completely unchanged. The FieldOperation approve path has no CropLot involvement.
