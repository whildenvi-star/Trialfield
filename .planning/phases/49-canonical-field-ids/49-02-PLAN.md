  <action>
Each script follows the same pattern. Build farm-budget first, then clone for fsa-acres with adjustments.

**Matching algorithm (implement in each script, not a shared module -- keep each script self-contained):**
1. Fetch all fields from farm-registry: `GET http://localhost:3005/api/fields`
2. Build a lookup map: for each registry field, normalize (trim + collapse multiple spaces to single + lowercase) the field name AND each alias. Map each normalized string to the registry field's `id`.
3. For each app record that has null/undefined registryFieldId:
   - Normalize the record's field name using the same function
   - Look up in the map
   - If exactly one match: mark as "matched" with the registry field ID
   - If zero matches: mark as "unmatched"
   - If multiple matches (ambiguous): mark as "ambiguous" with the candidate IDs (per user decision)
4. Records with an existing registryFieldId are skipped (idempotent).

**farm-budget/backfill-field-ids.js:**
- Load `data/data.json`
- Process `fields[]` array: match `field.name` to registry
- Process `rent[]` array: match `rent.fieldName` to registry
- Dry-run (default): print summary table (matched/unmatched/ambiguous/skipped counts) to console, write detailed report to `farm-budget/backfill-report.json` with arrays of {name, registryFieldId, status}
- With `--commit`: also write updated data.json with registryFieldId populated on matched records
- Coverage report includes: total records, matched count, unmatched list (names), ambiguous list, already-had-id count

**fsa-acres/backfill-field-ids.js:**
- Load `data/data.json`
- Process `cluRecords[]` array: match `record.fieldName` to registry
- Same dry-run/commit pattern
- Note: fsa-acres field names are lowercase (e.g., "daun" not "Daun") -- the case-insensitive matching handles this
- Write report to `fsa-acres/backfill-report.json`

Both scripts: `node backfill-field-ids.js` (dry run) or `node backfill-field-ids.js --commit` (write changes).
Farm-registry must be running on port 3005 for the scripts to fetch field data.

**grain-tickets/backfill-field-ids.js:**
- Grain-tickets uses Prisma + PostgreSQL. The Farm model already has `registryId String?`.
- Load Prisma client: `const { PrismaClient } = require('@prisma/client'); const prisma = new PrismaClient();`
- Fetch registry fields from `http://localhost:3005/api/fields`
- Query all Farm records where registryId is null: `prisma.farm.findMany({ where: { registryId: null } })`
- Match each Farm.name against registry using the same normalize+lookup algorithm
- Dry-run: print summary, write report to `grain-tickets/backfill-report.json`
- With --commit: `prisma.farm.update({ where: { id: farm.id }, data: { registryId: matchedFieldId } })` for each match
- Disconnect prisma client at end

**glomalin-portal/scripts/backfill-field-ids.ts:**
- Portal uses Supabase. This is a TypeScript script run with `npx tsx`.
- Import createClient from @supabase/supabase-js
- Use NEXT_PUBLIC_SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY from env (service role bypasses RLS)
- Fetch registry fields from `http://localhost:3005/api/fields`
- Query clu_records where registry_field_id is null: `supabase.from('clu_records').select('*').is('registry_field_id', null)`
- Match each record's `field_name` against registry
- Dry-run: print summary, write report to `glomalin-portal/scripts/backfill-report.json`
- With --commit: update each matched record: `supabase.from('clu_records').update({ registry_field_id: matchedFieldId }).eq('id', record.id)`
- Batch updates in groups of 50 to avoid rate limits
- Run with: `cd glomalin-portal && npx tsx scripts/backfill-field-ids.ts` or `npx tsx scripts/backfill-field-ids.ts --commit`
