  <action>
Create three portal API routes that proxy pass writes to organic-cert:

**POST /api/mobile/passes/confirm** — Confirm a planned pass
- Auth: getMobileUser (bearer token), reject if no certUserId (needed for operatorId mapping)
- Body: `{ fieldId, passId, operationDate?, operatorCertUserId? }`
  - `fieldId`: farm-budget field ID (from crop-plans detail page)
  - `passId`: the pass ID from the crop-plans detail response (originally the farm-budget machinery ID)
  - `operationDate`: ISO date string, defaults to today if omitted
  - `operatorCertUserId`: organic-cert User ID for operator override, defaults to caller's certUserId
- Resolve fieldEnterpriseId: call fetchCertService GET /api/fields?registryId={fieldId} to find the organic-cert field by registryId, then fetch its current-year FieldEnterprise. The organic-cert field lookup approach: fetch all fields via GET /api/fields (already exists), find the one with matching registryId, then use the field's ID to find FieldEnterprise for the current crop year via GET /api/field-enterprises?fieldId={certFieldId}&cropYear={currentYear}. If no FieldEnterprise found, return 404 with clear message.
- Alternative simpler approach: since organic-cert has a direct Prisma API accessible via its Next.js routes, proxy the write by POSTing to organic-cert `/api/field-enterprises/{fieldEnterpriseId}/operations` with the resolved fieldEnterpriseId. Pass: `{ type: passType, operationDate, operatorId: certUserId, passStatus: "CONFIRMED", plannedSource: "mobile-logger", budgetImplementId: passId, description: "Confirmed via mobile logger" }`
- Return: `{ success: true, fieldOperationId: string, confirmedAt: string }`
- On upstream failure: return 502 with `{ error: "organic-cert service unavailable" }`

**POST /api/mobile/passes/add** — Add an unplanned pass
- Auth: same as confirm
- Body: `{ fieldId, operationType, operationDate?, notes?, operatorCertUserId? }`
  - `operationType`: one of TILLAGE, PLANTING, SPRAYING, HARVEST, OTHER (map from the UI's friendly names — Herbicide/Fertilizer map to SPRAYING, Scouting maps to OTHER)
  - `operationDate`: defaults to today
  - `notes`: optional description
- Same fieldEnterpriseId resolution as confirm
- POST to organic-cert `/api/field-enterprises/{fieldEnterpriseId}/operations` with: `{ type: operationType, operationDate, operatorId: certUserId, passStatus: "CONFIRMED", plannedSource: "mobile-logger", dataSource: "MANUAL", description: notes }`
- Return: `{ success: true, fieldOperationId: string, pass: { id, type, status, operationDate, operatorName } }`

**PUT /api/mobile/passes/[passId]** — Edit a confirmed pass (date/operator)
- Auth: same
- Body: `{ fieldEnterpriseId, operationDate?, operatorCertUserId? }`
- Proxy PUT to organic-cert `/api/field-enterprises/{fieldEnterpriseId}/operations` with: `{ id: passId, operationDate, operatorId: operatorCertUserId }`
- Return: `{ success: true }`

**Mapping note for operation types:** The CONTEXT.md lists operation types as: Tillage, Planting, Herbicide, Fertilizer, Harvest, Scouting, Other. The organic-cert FieldOpType enum is: TILLAGE, PLANTING, CULTIVATION, MOWING, IRRIGATION, FLAMING, HARVEST, SPRAYING, OTHER. Create a mapping constant: Herbicide -> SPRAYING, Fertilizer -> SPRAYING, Scouting -> OTHER, the rest map directly to their uppercase equivalents. Export this mapping so Plan 02 can use it.

**Helper function:** Create a shared helper `resolveFieldEnterpriseId(fieldId: string): Promise<string>` in a new file `glomalin-portal/src/app/api/mobile/_lib/cert-bridge.ts` that:
1. GETs organic-cert fields list, finds field with registryId matching the farm-budget fieldId
2. GETs field enterprises for that field filtered by current crop year (new Date().getFullYear())
3. Returns the first matching fieldEnterpriseId or throws with a clear error
4. Also export the operation type mapping constant from this file
