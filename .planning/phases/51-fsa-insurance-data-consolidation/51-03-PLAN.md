  <name>Task 1: Create portal RMA price scraper API route</name>
  <files>glomalin-portal/src/app/api/insurance/pricing/scrape/route.ts</files>
  <action>
Create a POST API route at `/api/insurance/pricing/scrape` that:

1. **Auth gate**: Use `requireModuleAccess('insurance')` guard (same pattern as other insurance routes).

2. **Determine scope from CLU records**: Query `supabase.from('clu_records').select('crop')` to get distinct crops currently in the system. This makes the scraper automatically adapt to what crops are planted (per user decision).

3. **Fetch from USDA RMA**: Port the existing scraper logic from fsa-acres/server.js lines 307-353. The URL is:
   `https://public-rma.fpac.usda.gov/apps/PriceDiscovery/Services/RevenuePriceDataService.svc/RevenuePrices?discoveryPeriodDate={date}`
   - Use native `fetch()` (Next.js server, not the `https` module used in Express)
   - Parse the response JSON — items are in `data.d` or `data` array
   - Each item has: CommodityName, ProjectedPrice, HarvestPrice

4. **Update insurance_pricing table**: For each scraped price that matches a crop in clu_records:
   - Find matching row in `insurance_pricing` by crop name (case-insensitive)
   - If found and `manual_override` is not true, update `spring_price` (from ProjectedPrice) and `fall_price` (from HarvestPrice) and set `last_scraped` to ISO timestamp
   - If not found, insert a new pricing row with the scraped values
   - Use `supabase.from('insurance_pricing').upsert()` with crop as the conflict key

5. **Response**: Return `{ ok: true, updated: N, total: M, message: "N prices updated from USDA RMA", lastScraped: isoTimestamp }`

6. **Error handling**: If RMA is unreachable or returns bad data, return `{ ok: false, error: "reason", message: "details" }` with status 200 (not 500, so the UI can show the error message gracefully). Per user decision: keep last successful prices, never clear existing data on failure.

Note: Daily cron job is deferred — the manual trigger is the priority. Cron can be added later via Supabase scheduled functions or a simple setInterval in the portal server. Document this as a TODO comment in the route file.
