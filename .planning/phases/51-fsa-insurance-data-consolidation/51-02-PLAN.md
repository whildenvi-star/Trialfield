  <name>Task 2: Rewire server.js from JSON file to Supabase</name>
  <files>fsa-acres/server.js</files>
  <action>
This is the core task. Rewrite fsa-acres/server.js to use Supabase instead of the in-memory JSON store. The approach: replace the `store` object reads/writes with Supabase queries while keeping the same Express API contract (same request/response shapes).

**Architecture:**
- Initialize Supabase client at startup using `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` env vars
- If env vars are missing, log error and exit (same pattern as migration script)
- Remove `loadData()`, `saveData()`, `withLock()`, and the `store` object
- Keep `Calc` import — rollup/validation/insurance computations still happen server-side using data fetched from Supabase
- Settings (year, county, state, producerName) can remain in a small local JSON file (`data/settings.json`) since they're app config not farm data. Or store in Supabase — use Claude's discretion for least disruption.

**Column mapping** (fsa-acres camelCase → Supabase snake_case):
- CLU records: id→id (UUID), farmNumber→farm_number, tractNumber→tract_number, clu→clu, fieldName→field_name, farmName→farm_name, fsaAcres→fsa_acres, crop→crop, irrigated→irrigated, organic→organic, doubleCrop→double_crop, coverCrop→cover_crop, grainPlantDate→grain_plant_date, use→use, reported→reported, lineNumber→line_number, policyNumber→policy_number, unitNumber→unit_number, aph→aph, registryFieldId→registry_field_id, legacy_id→legacy_id
- Note: Some fields in fsa-acres JSON don't exist in Supabase schema (cc2023Species, cc2023PlantDate, tillage2024, cc2024, cc2024PlantDate, ntAdoption2024, ccAdoption2024, tillage2025, cc2025, cc2025PlantDate, ntAdoption2025, ccAdoption2025). These conservation practice tracking fields need to be handled. Options: (a) add columns to Supabase, (b) store in a JSONB metadata column, (c) drop them. Per user decision "no user-visible regression" — these fields appear in reports and exports, so they MUST be preserved. Add a `metadata` JSONB column to clu_records if it doesn't exist, or add the specific columns. Claude's discretion — JSONB metadata is cleanest since these are year-specific conservation fields that change format.
- Insurance: id→id, farmName→farm_name, farmNumber→farm_number, lineNumber→line_number, crop→crop, plantedAcres→planted_acres, guarantee→guarantee, actual→actual, claimStatus→claim_status, notes→notes, policyNumber→policy_number, coverageLevel→coverage_level, unitType→unit_type, premiumPerAcre→premium_per_acre, agentName→agent_name, policyYear→policy_year, preventedPlanting→prevented_planting, preventedPlantingAcres→prevented_planting_acres, etc.
- Pricing: id→id, crop→crop, springPrice→spring_price, fallPrice→fall_price, manualOverride→manual_override, lastScraped→last_scraped

**Endpoint-by-endpoint rewrite:**

1. **GET /api/clu-records** — Query `supabase.from('clu_records').select('*')` with optional `.eq()` filters for farmNumber, crop, fieldName, reported. Map snake_case response to camelCase for client compatibility.

2. **GET /api/clu-records/:id** — `supabase.from('clu_records').select('*').eq('id', id).single()`

3. **POST /api/clu-records** — `supabase.from('clu_records').insert({...mapped}).select().single()`. Generate `legacy_id` using the same generateId('clu') pattern for backward compatibility.

4. **PUT /api/clu-records/bulk** — For each ID in the array, `supabase.from('clu_records').update(mapped_updates).eq('id', id)`. Or use a single update with `.in('id', ids)` if all get the same updates.

5. **PUT /api/clu-records/:id** — `supabase.from('clu_records').update(mapped).eq('id', id).select().single()`

6. **DELETE /api/clu-records/:id** — `supabase.from('clu_records').delete().eq('id', id)`

7. **POST /api/clu-records/:id/split** — Fetch original, compute new acres, update original via Supabase, insert new record via Supabase. Same business logic, different persistence.

8. **POST /api/clu-records/:id/duplicate** — Similar: fetch, compute next CLU number from Supabase query, insert new.

9. **Rollup endpoints** (by-farm, by-crop, by-field, by-tract, tillage-summary, cover-crop-summary, summary-metrics, reporting-progress) — Fetch all CLU records from Supabase, then pass to existing Calc functions. Cache the fetch for 10 seconds to avoid hammering Supabase on rapid rollup calls.

10. **GET /api/pricing** — `supabase.from('insurance_pricing').select('*')`

11. **PUT /api/pricing/:id** — `supabase.from('insurance_pricing').update(mapped).eq('id', id)`

12. **POST /api/pricing** — `supabase.from('insurance_pricing').insert(mapped)`

13. **DELETE /api/pricing/:id** — `supabase.from('insurance_pricing').delete().eq('id', id)`

14. **POST /api/pricing/scrape** — Keep the USDA RMA HTTPS scraping logic, but write results to Supabase instead of store. This stays in fsa-acres for now (Plan 03 adds it to portal too, but fsa-acres keeps its version as a fallback).

15. **GET /api/insurance** — Fetch policies from Supabase, fetch CLU records + pricing for Calc.computeInsurancePolicy enrichment.

16. **Insurance CRUD** (GET/:id, POST, PUT/:id, DELETE/:id) — Standard Supabase operations with column mapping.

17. **GET /api/clu-aph** — Fetch CLU records from Supabase with crop filter, compute average APH.

18. **Validation, exports, cropping-intentions, sync-crops endpoints** — Fetch data from Supabase, pass to existing Calc/logic functions. These are read-heavy and don't need special treatment.

19. **Season dashboard** — Fetch CLU records + insurance from Supabase instead of store. Cross-app fetches (farm-budget, seed-inventory, grain-tickets, farm-registry) stay as-is.

20. **Settings** — Store in `data/settings.json` (small config file, not farm data). Create a minimal loadSettings/saveSettings pair.

21. **Farm numbers, field names, crop names** — Query distinct values from Supabase.

**Helper function: mapToClient(row)**
Create a helper that maps Supabase snake_case rows to camelCase for API responses, preserving backward compatibility with the frontend JS files.

**Helper function: mapToDb(body)**
Create a helper that maps camelCase request body to snake_case for Supabase writes.

**Error handling:**
Per user decision: if Supabase is unreachable, show clear error message. Writes are blocked. For reads, return 503 with `{ error: 'Data store unavailable' }`.

**Cache layer:**
Add a simple in-memory cache (10s TTL) for CLU records fetched from Supabase. Rollup endpoints call getCluRecords() which returns cached or fetches fresh. Cache is invalidated on any write operation. This prevents N+1 Supabase calls when the dashboard loads multiple rollups.

**CRITICAL: Do NOT modify any public/*.js files.** The API response shapes must remain identical so the existing vanilla JS frontend works without changes.
