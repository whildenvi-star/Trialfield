**2. Create glomalin-portal/src/lib/offline/sync-engine.ts:**

This module handles replaying queued operations against the API.

```
requestBackgroundSync(tag?: string): void
```
- Checks `navigator.serviceWorker?.ready` then calls `registration.sync.register(tag ?? 'pass-sync')`
- Silently catches if Background Sync not supported (falls back to manual sync)

```
processQueue(getToken: () => Promise<string | null>): Promise<SyncResult>
```
- Gets all pending operations from `offlineQueue.getPending()` — sorted by createdAt ascending (FIFO)
- Processes sequentially (one at a time, per user decision)
- For each operation, calls `replayOperation(op, token)`
- On success: delete the operation from queue via `offlineQueue.delete(op.id)`
- On transient error (HTTP 500, timeout, network error): increment retryCount, set status to 'failed' if retryCount >= 3, otherwise leave as 'pending'
- On 409 Conflict or detection that pass already confirmed: delete the operation, add to `skipped` list with reason "Already confirmed — skipped"
- On auth error (401): attempt token refresh via getToken(), retry once. If refresh fails, set ALL remaining items to 'failed' with errorMessage "Session expired — sign in to sync"
- On other client error (4xx): set status 'failed' with server's error message
- Returns `SyncResult { synced: number, skipped: SyncSkip[], failed: SyncFailure[], total: number }`
- Exponential backoff between retries: 1s, 4s, 16s (1000 * 4^retryCount)

```
replayOperation(op: QueuedOperation, token: string): Promise<ReplayResult>
```
- For type 'confirm-pass': call `POST /api/mobile/passes/confirm` with `{ fieldId: op.fieldId, passId: op.passId, passType: op.passType, operationDate: op.operationDate, operatorCertUserId: op.operatorId }`
- For type 'add-pass': call `POST /api/mobile/passes/add` with `{ fieldId: op.fieldId, operationType: op.operationType, operationDate: op.operationDate, notes: op.description, operatorCertUserId: op.operatorId }`
- Use AbortSignal.timeout(10000) on fetch calls
- Returns `{ status: 'synced' | 'conflict' | 'error', errorMessage?: string, httpStatus?: number }`

Export types: `SyncResult`, `SyncSkip`, `SyncFailure`, `ReplayResult`

**3. Extend sw.ts with Background Sync event listener:**

Add a `sync` event listener after `serwist.addEventListeners()`:

```ts
self.addEventListener('sync', (event: SyncEvent) => {
  if (event.tag === 'pass-sync') {
    event.waitUntil(handlePassSync())
  }
})
```

The `handlePassSync` function inside sw.ts:
- Opens IndexedDB directly (service worker cannot import from app modules that use `idb` library — use raw IndexedDB API)
- Opens DB 'glomalin-offline', reads all records from 'operation-queue' store where status === 'pending'
- For each pending op (sorted by createdAt), makes fetch() call to the appropriate endpoint (same logic as replayOperation but using raw fetch in SW context)
- On success: delete from store
- On conflict (409 or pass already CONFIRMED in response): delete from store
- On transient failure: increment retryCount, mark as 'failed' if >= 3 retries
- Note: service worker has no auth token access — it reads from a 'sync-meta' key in the operation-queue store (or a separate 'sync-config' object store). The client must store the current auth token in IndexedDB before registering the sync. Add a `setSyncToken(token: string)` export to sync-engine.ts that writes to a 'sync-config' store.

Update the DB version in db.ts from 2 to 3, adding a 'sync-config' object store in the upgrade handler (keyPath: 'key'). This store holds `{ key: 'auth-token', value: string }`.

Declare `SyncEvent` interface in sw.ts (TypeScript doesn't include it by default):
```ts
interface SyncEvent extends ExtendableEvent {
  tag: string
}
```
