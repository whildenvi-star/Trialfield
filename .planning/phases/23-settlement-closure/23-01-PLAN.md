<objective>
Add per-crop weight discrepancy tolerance configuration to the grain-tickets reconciliation system. Users can set a tolerance per crop (as percentage or fixed lbs) that controls when a weight match is flagged as a discrepancy vs acceptable variance.

Purpose: Without configurable tolerance, every weight difference shows as a discrepancy. Different crops have different acceptable shrink ranges — corn vs rye vs soybeans all behave differently. This lets the user tune the system to match their real-world expectations.

Output: CropConfig schema extension, tolerance CRUD API, tolerance settings UI, tolerance-aware reconciliation engine.
</objective>

<task type="auto">
  <name>Task 1: Add tolerance fields to CropConfig and create tolerance CRUD API</name>
  <files>grain-tickets/prisma/schema.prisma, grain-tickets/server.js</files>
  <action>
1. **Schema migration** — Add two new fields to the `CropConfig` model in `grain-tickets/prisma/schema.prisma`:
   - `tolerancePct Float @default(0)` — Percentage-based tolerance (e.g., 1.5 means 1.5%). When > 0, a matched ticket's weight variance up to this % is considered acceptable.
   - `toleranceLbs Float @default(0)` — Fixed-lbs tolerance (e.g., 500 means 500 lbs). When > 0 and tolerancePct is 0, this fixed amount is used instead.
   - Precedence rule: if both are > 0, use `tolerancePct` (percentage wins). If both are 0, every discrepancy is flagged (current behavior).
   Run `npx prisma migrate dev --name add-crop-tolerance` from `grain-tickets/` directory.

2. **Tolerance API endpoints** in `grain-tickets/server.js`:
   - `GET /api/crop-config/tolerances?cropYear=N` — Returns all CropConfig rows for a given cropYear with id, cropName, tolerancePct, toleranceLbs. Sort by cropName ascending.
   - `PUT /api/crop-config/:id/tolerance` — Accepts `{ tolerancePct, toleranceLbs }` body. Validates both are non-negative numbers. Updates the CropConfig row. Returns the updated row.
   Place these endpoints near the existing CropConfig routes (around line 100-200 area of server.js).

3. **Update `runMatch()` function** (around line 231 in server.js) to be tolerance-aware:
   - After matching by normalized ticket number, if a match is found AND both the farm ticket and settlement line have netWeight values, compute the weight variance: `Math.abs(farmTicket.netWeight - line.netWeight)`.
   - Look up the CropConfig for the farm ticket's crop + cropYear. Get tolerancePct and toleranceLbs.
   - Compute the threshold: if tolerancePct > 0, threshold = farmTicket.netWeight * (tolerancePct / 100); else if toleranceLbs > 0, threshold = toleranceLbs; else threshold = 0 (flag everything).
   - If variance > threshold AND threshold > 0, set matchStatus to 'matched' but also set a new field `discrepancyFlag` (Boolean, added to SettlementLine model) to true.
   - Actually, to avoid an extra migration: instead of adding a new DB field, include discrepancy info in the reconciliation summary response. The runMatch engine keeps matchStatus as 'matched' — the summary endpoint will compute tolerance violations on read.

4. **Update `GET /api/reconciliation/summary`** (around line 1646) to include tolerance checking:
   - After building the summary rows, for each crop row, look up the CropConfig tolerance for that crop+cropYear.
   - Compute the threshold (same logic: pct or lbs).
   - Add a `withinTolerance` boolean field to each summary row: true if `Math.abs(varianceLbs) <= threshold`, false otherwise.
   - Add `tolerancePct` and `toleranceLbs` values to each row for display context.
   - This avoids schema changes to SettlementLine — tolerance is evaluated at read time.
